## 1. HTTP/2 を学んだあとに何が見えるか

**初学者**  
ここまで読んで、HTTP/2 の形はだいぶ見えてきました。  
でも最初の目的の一つに、gRPC や HTTP/3 への橋渡しがありましたよね。  
HTTP/2 を学ぶと、それらにどうつながるんでしょうか。

**解説者**  
とても大事な問いです。  
この章では、HTTP/2 をゴールとして閉じるのではなく、次の技術を見るための土台として整理します。

**実装者**  
実際、gRPC や HTTP/3 を学ぶときも、HTTP/2 で身につけた観察のしかたがかなり役に立ちます。  
全部がそのまま同じではありません。  
でも、構造で見る視点は引き継げます。

**初学者**  
構造で見る視点、ですか。

**解説者**  
はい。  
この本で見てきたのは、HTTP/2 を「ただ速い HTTP」としてではなく、通信を単位に分けて理解することでした。  
接続全体を見る。  
個別の Stream を見る。  
その中を流れる Frame を見る。  
さらに、ログの中でその流れを観察する。  
この見方が、次の学習でも土台になります。

---

## 2. gRPC にどうつながるか

**初学者**  
まず gRPC ですが、これは HTTP/2 とどう関係するんですか。

**解説者**  
入門的には、gRPC は HTTP/2 の上に乗って動く RPC の仕組みと考えるとわかりやすいです。

**初学者**  
HTTP/2 を使って、プログラム同士がやり取りする仕組みなんですね。

**解説者**  
そうです。  
HTTP/2 が持っている Stream や多重化の仕組みを利用して、その上でリモートの関数やメソッドを呼ぶような通信を行います。

**実装者**  
だから gRPC を見るときにも、Stream の感覚が重要です。  
1回の呼び出しがどの Stream に対応しているのか。  
その中で、どんな順番で情報が流れているのか。  
こうした見方が役に立ちます。

**初学者**  
HTTP/2 を知らないと、その呼び出しの流れも見えにくいんですね。

**解説者**  
そうです。  
入門段階では、「1回の呼び出しを 1つの Stream として見る」とかなり整理しやすくなります。  
もちろん実際には gRPC 固有の約束もあります。  
ですが、HTTP/2 の Stream の上で通信しているという感覚があると、理解が安定します。

**初学者**  
でも gRPC って、普通の Web API とは少し雰囲気が違いますよね。

**解説者**  
そこも大事です。  
gRPC は HTTP/2 を使いますが、目的はブラウザで直接触る Web ページではなく、プログラム同士の効率的な通信です。  
そのため、HTTP/1.1 時代の REST API の見た目とはかなり違って感じられます。

**実装者**  
ただ、土台の観察ポイントは共通です。  
接続の上に Stream があり、その中にデータが流れる。  
この構造が見えていれば、gRPC もまったく未知のものには見えにくくなります。

## 3. HTTP/3 にどうつながるか

**初学者**  
次に HTTP/3 ですが、これは HTTP/2 の次世代版みたいな理解でいいんでしょうか。

**解説者**  
大まかにはそう言えます。  
ただし、HTTP/2 の上位互換とだけ思うと少し危険です。  
目的の一部は近いですが、下の層の作りはかなり違います。

**初学者**  
どこが大きく違うんですか。

**解説者**  
大きな違いは、HTTP/2 が TCP の上で動くのに対して、HTTP/3 は QUIC の上で動くことです。

**実装者**  
この違いはかなり大きいです。  
ただ、この本では QUIC の詳細には入りません。  
ここで押さえたいのは、HTTP/3 でも複数の流れを扱うが、その土台となる仕組みは HTTP/2 と同じではない、という点です。

**初学者**  
では、HTTP/2 を学んでも、そのまま使えるわけではないんですね。

**解説者**  
そのまま一致するわけではありません。  
でも、HTTP/2 を学ぶことはとても重要です。  
なぜなら、HTTP/3 を見たときにも、何を観察すればよいかがわかるからです。

**初学者**  
何を観察すればよいか、ですか。

**解説者**  
はい。  
たとえば、接続に相当するものは何か。  
Stream に相当するものはどう扱われるか。  
データはどんな単位で流れるか。  
多重化はどう見えるか。  
ログでは何を手がかりにするか。  
こうした問いを立てられるようになります。

**実装者**  
HTTP/2 を学んでいないと、HTTP/3 を見たときに全部が新しく見えすぎます。  
でも HTTP/2 を通っていれば、「これは何を置き換えた概念なんだろう」と比較しながら考えられます。

## 4. 次の学習で何を見ればよいか

**初学者**  
この本を読み終わったあと、次は何を学ぶとよいですか。

**解説者**  
方向は大きく二つあります。  
一つは gRPC 側に進む道です。  
HTTP/2 の上で、アプリケーション通信がどう設計されるかを見る道です。  
もう一つは HTTP/3 と QUIC 側に進む道です。  
HTTP の土台そのものがどう進化しているかを見る道です。

**初学者**  
どちらに進んでも、HTTP/2 の学習は無駄にならないんですね。

**解説者**  
その通りです。  
この本で育てたかったのは、単なる暗記ではなく、区別する力だからです。

**初学者**  
区別する力、というと。

**解説者**  
たとえば、Connection と Stream を区別すること。  
Stream と Frame を区別すること。  
本体の通信と制御の通信を区別すること。  
一つの流れと、複数の流れの重なりを区別することです。

**実装者**  
この力があると、新しいプロトコルやミドルウェアを見たときにも、「何が全体で、何が個別で、何が部品か」を探せるようになります。  
ログを見る。  
単位を区別する。  
流れを追う。  
パターンとして理解する。  
この四つは、そのまま次の学習にも持っていけます。

**初学者**  
最初は HTTP/2 って難しい規格の塊に見えていました。  
でも今は、接続の中で複数の Stream が流れて、その中を Frame が組み立てている、という形で見えるようになってきました。

**解説者**  
それがこの本の到達点です。  
すべての RFC を暗記することではありません。  
構造を言葉で説明できること。  
そして、ログを見て「今何が起きているか」を追えることです。

## 整理

この本で身につけた見方は、次のように整理できます。

- Connection / Stream / Frame を区別する
- HEADERS / DATA を区別する
- 本体の通信と制御の通信を区別する
- ログの流れを単位ごとに追う
- 複数の流れをパターンとして理解する

この見方は、次の学習にもつながります。

- gRPC  
  - HTTP/2 の Stream の上で呼び出しを考える
  - HEADERS と DATA の流れとして観察する

- HTTP/3  
  - HTTP/2 と同じではないことを前提にする
  - 接続相当、Stream 相当、データの流れを比較しながら見る

## この章の核心

HTTP/2 を学ぶ意味は、仕様項目を暗記することではありません。  
通信を単位に分け、流れとして観察し、全体と個別を区別しながら読む力を身につけることです。  
その見方があると、gRPC や HTTP/3 のような次の技術も、まったく新しいものとしてではなく、比較しながら理解できるようになります。

## まとめ

- HTTP/2 の理解は、gRPC や HTTP/3 を学ぶ土台になる
- gRPC では、HTTP/2 の Stream を意識した見方が役に立つ
- HTTP/3 は HTTP/2 と目的の一部を共有するが、土台の仕組みは同じではない
- この本で重要なのは、単位と流れを区別して観察する力である
- その力は、新しいプロトコルやミドルウェアを学ぶときにも使える

HTTP/2 入門は、ここで終わりです。  
ですが、ここで身につけた「区別して観察する力」は、この先のプロトコル学習でもずっと役に立ちます。
