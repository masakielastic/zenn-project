## 1. なぜログはバラバラに見えるのか

**初学者**  
第3章でフレームの種類はわかってきました。  
でも、実際の通信がどういう順番で流れるのかは、まだよく見えていません。  
ログを見ると、ただバラバラに並んでいるように見えてしまいます。

**解説者**  
そこが次の壁です。  
HTTP/2 はフレームという部品でできていますが、実際の通信ではそれらが一定の並び方をします。  
この章では、その並びを「通信パターン」として整理します。

**実装者**  
ログが読みにくい原因の多くは、順番の感覚がないことです。  
フレームの種類を知っていても、「次に何が来やすいか」が見えていないと、ただの羅列に見えてしまいます。

**初学者**  
つまり、フレームの名前を知るだけでは足りないんですね。

**解説者**  
その通りです。  
この章では、まず通信の本体に絞って、シンプルな形でパターンを見ます。  
制御のためのフレームはあとで別に扱います。

---

## 2. H と D で基本パターンを見る

**解説者**  
複雑さを減らすために、ここでは簡略記号を使います。  
H は HEADERS、D は DATA です。

**初学者**  
かなり単純になりますね。

**解説者**  
はい。  
HTTP メッセージの本体を観察するだけなら、まずはこの2つで十分です。  
HEADERS は条件を示す部分です。  
DATA は本文を運ぶ部分です。  
この章では、まず H と D の並びだけを追います。

**実装者**  
実際のログには SETTINGS や PING も出てきます。  
でも最初から全部を同時に追うと混乱しやすいです。  
だから最初は、HTTP メッセージの中心になる HEADERS と DATA に絞るのがよいです。

**初学者**  
まず本体だけを見て、あとで制御を足すわけですね。

**解説者**  
そうです。  
順番に複雑さを戻していきます。

---

## 3. 1つの Stream ではどう並ぶか

**初学者**  
では、1つの Stream だけを見ると、どんな並びになるんですか。

**解説者**  
まず、GET のようにリクエスト本文がない場合を考えましょう。  
そのとき、リクエスト側は HEADERS だけで終わることが多いです。  
一方でレスポンス側は、HEADERS のあとに DATA が続くことがよくあります。

**初学者**  
リクエストは H だけ。  
レスポンスは H のあとに D、ということですね。

**解説者**  
はい。  
これが最初に押さえたい基本形です。

GET の典型例

```text
リクエスト: H
レスポンス: H → D
```

**解説者**  
次に、POST や PUT のようにリクエスト本文がある場合です。  
このときは、リクエスト側にも DATA が続きます。  
レスポンス側にも本文があれば、同じように HEADERS のあとに DATA が続きます。

本文ありの典型例

```text
リクエスト: H → D
レスポンス: H → D
```

**初学者**  
リクエストにもレスポンスにも本文があれば、両方で D が出てくるんですね。

**解説者**  
その通りです。  
ただし、ここで大事なのは、DATA は1回だけとは限らないことです。

**初学者**  
分割される、という話ですね。

**解説者**  
はい。  
本文が大きければ、複数の DATA フレームに分かれて流れます。

DATA が分割される例

```text
レスポンス: H → D → D → D
```

**実装者**  
ログでも、DATA が何回も続くことはよくあります。  
だから D を1回見ただけで「本文は全部終わった」とは限りません。  
同じ Stream の中で、まだ続きがあるかを見ます。

**初学者**  
なるほど。  
1つの Stream だけなら、かなり素直な並びですね。

**解説者**  
そうです。  
HTTP/2 の通信は複雑に見えますが、1つの Stream の中だけを見ると、まずは H と D の並びとしてかなり単純に捉えられます。

## 4\. 複数の Stream が混ざるとどう見えるか

**初学者**  
でも実際のログでは、こんなにきれいには見えませんよね。

**実装者**  
そこが HTTP/2 らしいところです。  
現実のログでは、複数の Stream のフレームが交互に流れます。

たとえば、接続全体では次のように見えることがあります。

```text
stream1: H
stream3: H
stream1: H
stream3: D
stream1: D
```

**初学者**  
これは急に難しく見えますね。

**解説者**  
見た目はそうです。  
でも、やることは単純です。  
Connection 全体の並びをそのまま眺めるのではなく、Stream ごとに分けて考えます。

**初学者**  
つまり、同じ stream\_id のものをまとめるんですね。

**解説者**  
その通りです。  
さっきの例なら、次のように整理できます。

```text
stream1: H → H → D
stream3: H → D
```

**初学者**  
ああ、そうすると急に読めます。  
見た目では混ざっていても、各 Stream の中ではパターンとして理解できるんですね。

**実装者**  
そこが重要です。  
HTTP/2 のログは複雑に見えますが、実際には「複数の単純な流れが同時に進んでいる」と考えると読みやすくなります。

**解説者**  
第2章で見た多重化が、まさにこの見え方です。  
1本の Connection の上で、複数の Stream のフレームが入り混じって流れます。  
だから、ログを読むときは Connection 全体を見るだけでなく、必ず Stream 単位に戻って考える必要があります。

### 5\. 制御フレームをどう扱うか

**初学者**  
実際のログだと、SETTINGS や PING も混ざりますよね。  
あれはどう見ればいいんですか。

**解説者**  
大切なのは、制御フレームを邪魔なものと考えないことです。  
ただし、最初から本体と同じ重さで追わなくてもかまいません。  
まずは HTTP メッセージの本体を見て、そのあとで制御を足していけば十分です。

**実装者**  
たとえば、ログではこんな並びもあります。

```text
SETTINGS
stream1: H
WINDOW_UPDATE
stream1: H
stream1: D
PING
```

**初学者**  
確かに混ざっていますね。

**解説者**  
はい。  
このときの考え方は単純です。  
HEADERS と DATA は HTTP メッセージの本体です。  
それ以外は、接続設定、流量調整、状態確認などの制御です。  
まず本体の流れを追い、そのあとで制御がどこに入っているかを見ます。

**初学者**  
最初から全部を同時に読む必要はないんですね。

**解説者**  
そうです。  
本体と制御を分けて考えると、理解がかなり安定します。

**実装者**  
ログを見るときも同じです。  
最初は H と D の並びだけを見る。  
慣れてきたら、SETTINGS や WINDOW\_UPDATE や PING が、接続全体のどの場面で出ているかを見る。  
この順番が実践的です。

## 整理

この章で使った簡略記号は次の通りです。

- H = HEADERS
    
- D = DATA
    

1つの Stream だけを見れば、基本パターンはかなり単純です。

GET の典型例

```text
リクエスト: H
レスポンス: H → D
```

本文ありの典型例

```text
リクエスト: H → D
レスポンス: H → D
```

DATA 分割の例

```text
レスポンス: H → D → D → D
```

ログを読むときは、次の順番で見ると整理しやすくなります。

- まず、HEADERS と DATA に絞って本体を見る
    
- 次に、stream\_id ごとに分けて並びを見る
  
- DATA が複数続くなら、本文が分割されていると考える
  
- そのあとで、SETTINGS や PING などの制御フレームを見る

## この章の核心

HTTP/2 のログは複雑に見えますが、各 Stream を H と D の並びとして見れば、通信の流れはかなり単純に整理できます。  
複雑なのは見た目であり、中身は複数の単純な流れが同時に進んでいる構造です。

## まとめ

- HTTP/2 の通信は、まず H と D の並びとして単純化して見られる
    
- 1つの Stream だけを見ると、基本パターンは比較的素直である
    
- DATA は複数のフレームに分割されることがある
    
- 実際のログでは複数の Stream が混ざるため、stream\_id ごとに分けて読むことが重要である
    
- 制御フレームは本体とは分けて考えると理解しやすい
    

次の章では、実際のログを見ながら、ここで整理した読み方を試します。  
H と D の並びを意識して読むと、ログがただの羅列ではなく、意味のある流れとして見えてきます。
