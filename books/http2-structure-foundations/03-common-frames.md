## 1. まず全部覚えなくてよい

**初学者**  
第2章で Connection と Stream と Frame の違いはだいぶわかってきました。  
でも、Frame も種類がいろいろありますよね。  
全部覚えないといけないんでしょうか。

**解説者**  
入門の段階では、まず全部を覚えなくて大丈夫です。  
HTTP/2 にはいくつもフレームがありますが、最初は役割の大きいものから見れば十分です。

**初学者**  
役割の大きいもの、ですか。

**解説者**  
はい。  
まずは **HEADERS** と **DATA** を押さえます。  
この2つは、HTTP メッセージそのものを観察するときに中心になるからです。  
そのうえで、接続設定や状態確認、終了処理のためのフレームがある、と考えると整理しやすくなります。

**実装者**  
実際のログでも、最初に目につきやすいのは HEADERS と DATA です。  
接続の開始時には SETTINGS が出ますし、終了に近い場面では GOAWAY や RST_STREAM が見えることがあります。  
でも、読み始めでまず追いたいのは HEADERS と DATA です。

**初学者**  
やはり主役はその2つなんですね。

**解説者**  
そうです。  
この章では、まず HEADERS と DATA を中心に見ます。  
そのあとで、ほかの代表的なフレームを概観します。

---

## 2. HEADERS フレーム

**初学者**  
HEADERS は、名前の通りヘッダーを送るフレームですよね。

**解説者**  
はい。  
HEADERS フレームは、リクエストやレスポンスの条件を示す情報を運びます。

**初学者**  
条件というと、どんなものですか。

**解説者**  
リクエストなら、メソッドやパスのような情報です。  
レスポンスなら、ステータスや各種ヘッダーのような情報です。  
つまり、「この通信が何であるか」を先に示す部分だと考えるとわかりやすいです。

**初学者**  
HTTP/1.1 だと、最初の行とヘッダーがテキストで並んでいましたよね。

**解説者**  
そうです。  
HTTP/1.1 では、開始行とヘッダーが文字の並びとして見えやすかったです。  
HTTP/2 では、それが HEADERS フレームという部品に入って送られます。  
見た目は変わりますが、役割は近いです。

**実装者**  
ログでも、ある stream に対して最初に HEADERS が出ることが多いです。  
だから HEADERS を見たら、「この stream でどんな要求や応答が始まるのか」を示していると考えると読みやすいです。

**初学者**  
では、1つの stream は HEADERS から始まることが多いんですね。

**解説者**  
はい。  
HTTP メッセージとして意味のあるやりとりを始めるには、まずその内容を示す情報が必要です。  
その入口になるのが HEADERS です。

---

## 3. DATA フレーム

**初学者**  
DATA は本文ですね。

**解説者**  
その通りです。  
DATA フレームは、リクエスト本文やレスポンス本文の実データを運びます。

**初学者**  
HTML でも JSON でも画像でも、本文なら DATA なんですか。

**解説者**  
はい。  
HTTP の本文として運ばれるものは、基本的に DATA フレームに入って流れます。

**初学者**  
HTTP/1.1 だと、本文はそのまま後ろに続く感じでしたよね。

**解説者**  
HTTP/2 では、本文も小さな部品に分かれて送られます。  
そのため、本文が 1つの DATA フレームだけで終わるとは限りません。  
複数の DATA フレームに分かれて流れることがあります。

**初学者**  
では、DATA を1回見ただけでは本文全部とは限らないんですね。

**実装者**  
その通りです。  
ログでも、少し大きいレスポンスなら DATA フレームが何個か続くことがあります。  
だから、DATA が出たら終わりではなく、同じ stream の中で続きがあるかを見ることが大切です。

**解説者**  
ここで大事なのは、**1つの HTTP メッセージが 1つのフレームに対応するわけではない**ということです。  
リクエストやレスポンスは、HEADERS と DATA に分かれて流れます。  
そして DATA は、さらに複数に分かれることがあります。

---

## 4. HEADERS と DATA の基本パターン

**初学者**  
ここまで聞くと、HTTP/2 の基本パターンはかなり単純に見えてきます。

**解説者**  
はい。  
入門でまず覚えるべき形は、とてもシンプルです。  
最初に HEADERS が来ます。  
そして、本文が必要ならそのあとに DATA が続きます。

**初学者**  
リクエストでもレスポンスでも、その見方でよいですか。

**解説者**  
はい。  
もちろん細かい条件や例外はあります。  
ですが、最初の理解としてはそれで十分です。

**初学者**  
たとえば GET リクエストなら、本文がないことが多いですよね。

**解説者**  
そうです。  
その場合、リクエスト側は HEADERS だけで終わることがあります。  
一方でレスポンス側は、HEADERS のあとに DATA が続くことが多いです。

**実装者**  
ログを読むときも、この並びを探すとわかりやすいです。  
同じ stream_id の中で HEADERS があり、そのあとに DATA が続いていれば、「この stream で本文が流れているのだな」と読めます。

**初学者**  
つまり、HTTP/2 のログを見るときは、まず HEADERS と DATA の並びを探せばよいんですね。

**解説者**  
その通りです。  
ここが見えるようになると、通信の流れがかなり具体的に見えてきます。

## 5. そのほかの代表的なフレーム

**初学者**  
HEADERS と DATA 以外も、名前だけは少し気になります。

**解説者**  
では、ここでは代表的なものだけを大まかに見ましょう。  
大切なのは、HTTP メッセージの中身を運ぶフレームとは別に、接続や制御のためのフレームがあると知ることです。

**初学者**  
具体的にはどんなものがありますか。

**解説者**  
たとえば **SETTINGS** は、接続に関する設定をやりとりするフレームです。  
通信のかなり早い段階で出てくることが多く、個別のリクエストやレスポンスというより、接続全体の準備に関わります。

**実装者**  
ログで SETTINGS が早い段階に出たら、「いまは個別のページ取得というより、接続の準備の話をしている」と考えると読みやすいです。

**解説者**  
**WINDOW_UPDATE** は、どれだけ送ってよいかの調整に関わるフレームです。  
詳しい仕組みはあとで扱いますが、ここでは「データ量の調整に使う」と押さえれば十分です。

**解説者**  
**PING** は、接続確認や往復時間の確認に使われます。  
HTTP メッセージそのものではなく、接続が生きているかを確かめるための補助的なフレームです。

**解説者**  
**RST_STREAM** は、特定の stream を途中で打ち切るためのフレームです。  
接続全体ではなく、その 1件のやりとりだけを終わらせたいときに使われます。

**初学者**  
第2章の区別がここで効いてきますね。  
stream 単位で止めるのが RST_STREAM なんですね。

**解説者**  
その通りです。  
そして **GOAWAY** は、接続全体を閉じる方向に進むときのフレームです。  
こちらは stream 単位ではなく、connection 単位の話です。

**初学者**  
RST_STREAM は個別、GOAWAY は全体。  
この違いは覚えやすいです。

**実装者**  
ログでも、その違いは重要です。  
RST_STREAM が出たら「この stream が途中で終わった」と読めますし、GOAWAY が出たら「接続全体が終わりに向かっている」と読めます。

**解説者**  
入門の段階では、ここまでで十分です。  
まずは、HEADERS と DATA が HTTP メッセージの中心を担う。  
そのほかに、接続設定、状態確認、流量調整、終了処理のためのフレームがある。  
この二層で整理しておけば、最初の理解としては十分です。

---

## 整理

入門でまず押さえたいフレームは、次の2種類に分けられます。

- HTTP メッセージの内容を観察するときに中心になるもの  
  - HEADERS  
  - DATA

- 接続や制御のために使う代表的なもの  
  - SETTINGS  
  - WINDOW_UPDATE  
  - PING  
  - RST_STREAM  
  - GOAWAY

ログを読むときは、次の順番で見ると整理しやすくなります。

- まず、HEADERS や DATA のようなメッセージ中心のフレームかどうかを見る
- 次に、同じ stream_id の中で並びを追う
- そのうえで、制御系のフレームがどこで出ているかを見る

## この章の核心

HTTP/2 のフレームを最初から全部同じ重さで覚える必要はありません。  
まずは HEADERS がリクエストやレスポンスの条件を示し、DATA が本文を運ぶと理解することが重要です。  
そのうえで、接続や制御のためのフレームが別にあると押さえると、ログが読みやすくなります。

## まとめ

- 入門では、まず HEADERS と DATA を中心に見ればよい
- HEADERS はリクエストやレスポンスの条件を示す情報を運ぶ
- DATA は本文を運び、複数のフレームに分かれることがある
- SETTINGS や PING などは接続や制御のためのフレームである
- ログでは、フレームの種類と stream_id をあわせて見ると読みやすい

次の章では、これらのフレームが実際にどのような順番で並ぶのかを見ていきます。  
フレームの名前だけでなく、出てくる順番まで見えるようになると、HTTP/2 のログはさらに読みやすくなります。
