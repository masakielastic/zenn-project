## 1. なぜこの3つが紛らわしいのか

**初学者**  
HTTP/2を勉強し始めると、Connection と Stream と Frame が何度も出てきますよね。  
でも、正直ぜんぶ「通信の単位」に見えてしまいます。何が違うんでしょうか。

**解説者**  
そこは、HTTP/2の最初の山場です。  
逆に言うと、この3つの区別がつくようになると、HTTP/2はかなり見通しがよくなります。

**初学者**  
そんなに大事なんですね。

**解説者**  
はい。  
混乱しやすい理由は、3つとも通信に関係する言葉だからです。  
でも実際には、同じ大きさの概念ではありません。  
接続全体の話なのか。  
その中の個別のやりとりなのか。  
それとも、そのやりとりを構成する部品なのか。  
まずは、この階層の違いを意識することが大切です。

**実装者**  
ログを見ると、その違いがかなりはっきり出ます。  
接続全体に関する話と、特定の stream_id に関する話が混ざって出てくるからです。  
そこを分けて読めるようになると、急に分かりやすくなります。

**初学者**  
つまり、同じ「通信の話」でも、見ている大きさが違うんですね。

**解説者**  
その通りです。  
では、いちばん大きいところから順に見ていきましょう。

## 2. Connection とは何か

**解説者**  
まず Connection です。  
Connection は、クライアントとサーバーのあいだの1本の接続です。

**初学者**  
HTTP/1.1でも接続はありましたよね。

**解説者**  
はい。HTTP/1.1でもTCP接続はありました。  
HTTP/2でも、まずクライアントとサーバーが接続を作ります。  
それが Connection です。

**初学者**  
では、HTTP/2で特別なのはどこですか。

**解説者**  
大事なのは、HTTP/2ではこの1本の接続の上で、複数のやりとりを同時に進めやすいことです。  
HTTP/1.1でも接続の再利用はありましたが、HTTP/2では接続の使い方そのものが、より整理された形になっています。

**初学者**  
1本の接続をうまく共有する、という感じですか。

**解説者**  
そうです。  
HTTP/2では、接続をただの通り道として使うのではなく、その中に複数のやりとりを整理して載せる前提で設計されています。

**実装者**  
ログでも、まず「いま1本の接続の中の話をしている」と意識すると読みやすくなります。  
その上で、個別の stream_id を見ていくことになります。

---

## 3. Stream とは何か

**初学者**  
では、その接続の中の「複数のやりとり」が Stream なんですか。

**解説者**  
はい。  
Stream は、Connection の中で進む個別のやりとりです。

**初学者**  
個別のやりとり、ですか。

**解説者**  
たとえば、HTMLを取るリクエストが1つあるとします。  
CSSを取るリクエストがもう1つあります。  
さらに JavaScript を取るリクエストもあるとします。  
HTTP/2では、それぞれを別々の Stream として扱えます。

**初学者**  
ということは、1回のリクエストとレスポンスの組に近いんですね。

**解説者**  
入門段階では、その理解で大丈夫です。  
厳密には、Stream は HTTP/2 接続の中に作られる双方向の通信の流れです。  
ただ最初は、**接続の中の1件ごとのやりとり**と思っておくと整理しやすいです。

**初学者**  
Connection が全体で、Stream がその中の一件ずつ。  
少し見えてきました。

**実装者**  
ログでは、ここに stream_id が付きます。  
stream 1、stream 3、stream 5 のような番号が出てきたら、「いまどのやりとりの話をしているのか」を示していることが多いです。

**初学者**  
じゃあ、ログで stream_id を見ると、個別のやりとりを追えるんですね。

**実装者**  
そうです。  
HTTP/2のログを読むときに、最初に効いてくるのがそこです。

---

## 4. Frame とは何か

**初学者**  
では、Frame は何ですか。  
Stream とどう違うんでしょうか。

**解説者**  
Frame は、HTTP/2で実際に送られる最小の部品です。

**初学者**  
最小の部品、ですか。

**解説者**  
はい。  
HTTP/2では、リクエストやレスポンスを、大きなひとかたまりとして送るのではありません。  
用途ごとの小さな部品に分けて送ります。  
その部品が Frame です。

**初学者**  
たとえば、どんなものがありますか。

**解説者**  
代表的なものに HEADERS フレームと DATA フレームがあります。  
HEADERS フレームは、メソッドやパス、ステータス、各種ヘッダーのような情報を運びます。  
DATA フレームは、本文を運びます。

**初学者**  
つまり、1つの Stream の中に HEADERS や DATA が並ぶんですね。

**解説者**  
その通りです。  
たとえばリクエスト側では、まず HEADERS フレームでリクエストの情報を送り、必要ならそのあと DATA フレームで本文を送ります。  
レスポンス側でも、まず HEADERS フレームでステータスやレスポンスヘッダーが来て、そのあと DATA フレームで本文が来ます。

**初学者**  
なるほど。  
Stream はやりとりそのもので、Frame はそのやりとりを作っている部品なんですね。

**解説者**  
はい。  
ここで大切なのは、Connection、Stream、Frame は同じ階層の言葉ではない、ということです。  
Connection は全体です。  
Stream はその中の個別の流れです。  
Frame はさらにその中の部品です。

---

## 5. 多重化は何を意味するのか

**初学者**  
ここまで聞くと、HTTP/2が整理された仕組みに見えてきました。  
でも、実際には何がうれしいんですか。

**解説者**  
大きいのは、多重化できることです。  
つまり、複数の Stream を1本の Connection の上で並行して扱えることです。

**初学者**  
並行して扱う、というのは、フレームが入り混じって流れるということですか。

**解説者**  
そうです。  
たとえば、stream 1 の HEADERS が来たあとに、stream 3 の HEADERS が来ることがあります。  
そのあとに stream 1 の DATA が来て、さらに stream 5 の HEADERS が続くこともあります。  
接続全体で見ると、複数の Stream の Frame が交互に流れてよいのです。

**初学者**  
それで混乱しないんですか。

**解説者**  
混乱しないように、各 Frame には「どの Stream に属するか」という情報があります。  
受け取った側は、それを見て同じ Stream の部品を組み立てます。

**実装者**  
ログでも、その感覚がそのまま出ます。  
たとえば `HEADERS frame ... stream_id=3` のように見えれば、「これは stream 3 に属する HEADERS フレームだ」と読めます。  
だからログでは、Frame の種類と stream_id を追うことが基本になります。

**初学者**  
なるほど。  
HTTP/2が難しく感じるのは、全部を同じ「通信」として見てしまうからなんですね。

**解説者**  
その通りです。  
HTTP/2では、通信を階層に分けて整理している、と考えるほうが自然です。  
その整理があるから、1本の接続の上で複数のやりとりを無理なく扱えるのです。

---

## 整理

Connection、Stream、Frame の違いは、次のように整理できます。

- Connection：クライアントとサーバーのあいだの1本の接続
- Stream：その接続の中で進む個別のやりとり
- Frame：そのやりとりを構成する最小の部品

ログを読むときは、次の順番で見ると混乱しにくくなります。

- まず、接続全体の中の話をしていると意識する
- 次に、どの stream_id の話かを見る
- 最後に、HEADERS や DATA などの Frame の種類を見る

## この章の核心

HTTP/2では、Connection、Stream、Frame は同じ大きさの概念ではありません。  
Connection は接続全体、Stream はその中の個別のやりとり、Frame はそのやりとりを構成する最小の部品です。  
この階層を分けて考えることが、HTTP/2理解の土台になります。

## まとめ

- Connection はクライアントとサーバーのあいだの1本の接続である
- Stream はその接続の中で進む個別のやりとりである
- Frame は Stream を構成する最小の部品である
- 複数の Stream の Frame は1本の接続の上で交互に流れる
- ログでは Frame の種類と stream_id を追うと読みやすい

次の章では、Frame の中でも特によく出てくる HEADERS や DATA を取り上げます。  
そこまで分かると、HTTP/2の通信ログをより具体的に読めるようになります。
