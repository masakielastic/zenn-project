---
title: "Future は Promise の別名ではない"
---

**初学者**  
ここまで読むと、だんだん `Promise` を雑に使わない感覚は出てきました。  
でも次の章の題名を見ると、また少し混乱します。  
`Future` って、結局 `Promise` の別の呼び方ではないんですか。  
どちらも「あとで結果が出るもの」に見えます。

**解説者**  
その見え方は自然です。  
実際、入門的な説明では似た言葉として並べられることもあります。  
ただ、この本ではそこをはっきり分けます。  
この章の中心は、**`Promise` は結果側から見ると整理しやすく、`Future` は計算側から見ると整理しやすい**、という点です。

**実装者**  
JavaScript しか触っていないと、この違いはかなり見えにくいです。  
なぜなら日常では `Promise` が、結果への接続と、処理が進んでいる感じの両方をまとめて背負って見えるからです。

**初学者**  
まさにそこです。  
`Promise` で困っていないので、「`Future` って別名でしょう」で済ませたくなります。

**解説者**  
そこで済ませると、Rust や Python の async を読むときに急に足元が崩れます。  
この章では、名前の違いではなく、**責務の置き方の違い**として `Future` を見ていきます。

## Promise と Future は、どこが似て見えるのか

**初学者**  
まずは似て見える理由から確認したいです。

**解説者**  
よい入り方です。  
両方とも、最終的には「あとで結果にたどり着くもの」に見えます。  
そのため、表面だけ見るとかなり近いです。

**実装者**  
成功するかもしれない。  
失敗するかもしれない。  
いまはまだ完成していない。  
後で値が得られる。  
このあたりの印象は重なります。

**初学者**  
だから「同じ棚でいいでしょう」と思ってしまうわけですね。

**解説者**  
はい。  
でも、本書ではその棚を少し分けます。  
`Promise` は「結果の受け取り口」として見る。  
`Future` は「これから進められる計算」として見る。  
この区別を置くと、後で task や runtime が見えやすくなります。

## Future は「将来値」より「計算」として見る

**初学者**  
ここが核心ですね。  
`Future` は計算なんですか。

**解説者**  
この本では、そう捉えます。  
もちろん厳密な用語の使われ方は、言語や文脈で揺れます。  
ただ、責務分解のためには、`Future` を **まだ進行途中で、これから進められる計算** として見るのが有効です。

**初学者**  
将来値ではなく、進行途中の計算。

**解説者**  
はい。  
`Promise` を「結果が後で届く器」と見るのに対して、`Future` は「まだ終わっていない仕事の側」に重心を置いています。  
最終的に値にたどり着く点は似ています。  
ですが本書では、**その値へ向かう途中の進み方** に注目したいので、`Future` を計算として置きます。

**実装者**  
この違いは大きいです。  
`Promise` だと、呼び出し側は結果を待つ感覚が強い。  
`Future` だと、まだ計算の本体が残っていて、それをどのように前へ進めるのかという感覚が前に出てきます。

### 整理

- **Promise**  
    後で確定する結果への接続口として見る
    
- **Future**  
    まだ完了していない計算として見る
    
- 両者は似て見えるが、重心が違う
    
- この違いを置くと、実行基盤との関係が整理しやすくなる
    

**初学者**  
なるほど。  
`Promise` は出口寄り、`Future` は進行中の本体寄り、という感じですね。

**解説者**  
その表現でかなり近いです。

## 「誰が進めるのか」という問いが前に出てくる

**初学者**  
`Future` を計算として見ると、前の章までより実行モデルの話が前に出てきますね。

**解説者**  
そこが重要です。  
`Promise` を将来値として見るだけなら、「あとで受け取る」でかなりのところまで進めます。  
でも `Future` を計算として見ると、すぐに次の問いが出てきます。  
**その計算は、誰が進めるのか。**

**実装者**  
ここで runtime や executor の話につながってきます。  
計算があるだけでは完了しません。  
どこかで進める仕組みが必要です。

**初学者**  
JavaScript だと、この問いをあまり強く意識しないですね。  
`await` していれば、いつの間にか進んでいる感じがします。

**解説者**  
まさにそこです。  
JavaScript では、この問いがかなり隠れています。  
でも `Future` の考え方に入ると、「値」よりも「進める計算」が前に出るので、実行基盤の存在が見えやすくなります。

## poll という考え方

**初学者**  
ここで `poll` が出てくるんですね。  
名前だけは聞いたことがありますが、かなり抽象的です。

**解説者**  
`poll` は、この章でぜひ感覚をつかみたい言葉です。  
くだけて言えば、**「もう進められますか」と計算の状態を確かめること**です。

**初学者**  
計算に対して、「進められるか」を聞く感じですか。

**解説者**  
はい。  
`Future` を計算として見ると、いま完了しているとは限らない。  
まだ待つ必要があるかもしれない。  
少し進めたら、また保留になるかもしれない。  
そういう計算に対して、状態を確かめ、進められるなら進める。  
この感覚が `poll` です。

**実装者**  
ここでは、readiness の章で見た「準備ができたら進める」という感覚ともつながります。  
ただし今回は、I/O ハンドルそのものではなく、**計算の進行状態** に注目しています。

**初学者**  
つまり `poll` は、ソケットを直接読む話ではなく、計算の側を前に進める話なんですね。

**解説者**  
その通りです。  
この違いが見えると、`Future` は単なる将来値ではなく、進行中の仕事として見えてきます。  
そして、この「確かめる」働きは、計算そのものだけで完結する話ではありません。  
それを進める側の土台とも関係してきます。

### 整理

- `poll` は、計算がいま進められるかを確かめる発想
    
- 完了していれば結果に到達する
    
- まだなら保留され、後で再び進められる
    
- したがって `Future` は、受け取るだけの値ではなく、進行状態を持つ計算として見える
    

## executor と runtime は何をしているのか

**初学者**  
`Future` が計算なら、それを進める人が必要になる。  
そこで executor や runtime が出てくるんですね。

**解説者**  
はい。  
ここで前の章までの話がつながります。  
`Stream` や `Socket` はハンドルでした。  
`Event Loop` は実行基盤でした。  
`Future` は計算です。  
すると、その計算をどう進めるかという土台が必要になります。

**実装者**  
その土台として見えるのが executor や runtime です。  
言葉の使い分けは文脈で揺れますが、この本ではひとまずこう置きます。  
executor は、仕事を進める実務寄りの側面。  
runtime は、その仕事が動ける全体の土台。  
完全に別物として切りすぎなくてもよいですが、少なくとも `Future` 自体ではないことは大切です。

**初学者**  
ここでも、「計算」と「計算を進める土台」は別なんですね。

**解説者**  
その通りです。  
`Future` を見ているだけでは、まだ仕事は完了しません。  
進行には、調停や再開の仕組みが必要です。  
この視点がないと、後で task の話になったときに混乱します。

## task は何が違うのか

**初学者**  
ここでさらに `task` が出てくると、また混ざりそうです。  
`Future` と `task` はどう違うんでしょう。

**解説者**  
この章では、接続だけ押さえましょう。  
`Future` は計算として考える。  
`task` は、その計算が **実行基盤の上で仕事として扱われる単位** として考える。  
そんなふうに分けると理解しやすいです。

**実装者**  
くだけて言えば、  
`Future` は「こういう計算があります」  
`task` は「その計算をこの基盤で動かします」  
に近いです。  
ここまで見えてくると、寿命管理や親子関係、失敗の扱いといった話が、`Future` 単体ではなく仕事単位の話として見えやすくなります。

**初学者**  
なるほど。  
`Future` は計算の形、`task` は動いている仕事の単位、という感じですね。

**解説者**  
はい。  
もちろん言語によって見え方は違います。  
でも責務分解としては、その区別があると整理しやすいです。  
ただし、この章の主役はあくまで `Future` です。  
ここでは、`Future` が計算であり、それが基盤の上で仕事として扱われうる、という接続だけ押さえれば十分です。

### 整理

- **Future**  
    進行途中の計算として見る
    
- **executor / runtime**  
    その計算を進める土台として見る
    
- **task**  
    基盤の上で実際に扱われる仕事の単位として見る
    
- したがって、`Future = task = runtime` ではない
    

## await・spawn・join は同じ操作ではない

**初学者**  
ここまで来ると、`await` 以外の言葉も気になります。  
`spawn` や `join` はどう位置づければいいですか。

**解説者**  
この章では、細かいAPIではなく責務の違いだけ押さえましょう。  
`await`、`spawn`、`join` は、どれも非同期の文脈で出てきますが、同じ操作ではありません。

**実装者**  
`await` は、結果を待つ。  
`spawn` は、仕事を実行基盤に載せる。  
`join` は、複数の仕事の完了を合わせて待つ。  
ざっくり言えば、こんな違いがあります。

**初学者**  
これはたしかに大事ですね。  
今まで全部「非同期のための命令」くらいにまとめていました。

**解説者**  
そこを分けるのが本書の目的です。  
何かを待つのか。  
何かを起動するのか。  
何かを合流させるのか。  
この違いを無視すると、設計上の責任範囲がすぐ曖昧になります。

### 整理

- **await**  
    結果を待つ
    
- **spawn**  
    仕事を実行基盤に載せる
    
- **join**  
    複数の仕事の完了を合流させる
    
- 同じ非同期文脈に出るが、責務は異なる

**初学者**  
ここまでの話、かなり整理されてきました。  
でも同時に、JavaScript ユーザーがここで混乱しやすい理由もよくわかります。

言葉にしてみましょうか。

JavaScript では、`Promise` が結果にも見えるし、処理にも見える。  
`await` で待てるので、実行基盤も意識しにくい。  
そのせいで、`Future` のように「計算そのもの」に重心を置いた見方をしたとき、どこを分ければいいのかわからなくなる。  
そんな感じです。

**解説者**  
非常によい整理です。  
まさにその通りです。

**実装者**  
だからこの章では、無理に「JavaScript の Promise は間違いだ」とは言いません。  
そうではなく、JavaScript で圧縮されて見えていたものを、他言語でも読めるように分解しているんです。

**初学者**  
つまり、JavaScript の経験を捨てるのではなく、そこに新しいラベルを貼っているわけですね。

**解説者**  
はい。  
その結果として、次の章以降で task、runtime、責任スコープの話がしやすくなります。

## Promise と Future をどう使い分けて理解するか

**初学者**  
最後に確認したいです。  
この本では、`Promise` と `Future` をどう使い分けて理解すればいいでしょう。

**解説者**  
こう覚えるとよいです。  
`Promise` は、まず結果の側から見る。  
`Future` は、まず計算の側から見る。  
もちろん現実のライブラリや言語仕様では、境界がぴったり一致しないこともあります。  
でも責務分解の地図としては、この違いが非常に有効です。

**実装者**  
この地図があると、  
「これは何を待っているのか」  
「これは誰が進めるのか」  
「これは計算なのか、結果なのか、仕事単位なのか」  
を落ち着いて見分けられるようになります。

**初学者**  
ここまで来ると、`Future` は別名というより、別の角度から見た重要な概念だとわかってきました。

### この章の核心

`Future` は、`Promise` の単なる別名として見るよりも、進行途中の計算として見るほうが整理しやすいです。  
そうすると、`poll`、executor、runtime、task との関係が見えやすくなります。  
`Promise` を結果側、`Future` を計算側から見ることで、非同期処理の地図は一段はっきりします。

## まとめ

- `Promise` と `Future` は似て見えるが、重心の置き方が違う。
    
- `Promise` は将来値、`Future` は進行途中の計算として見ると整理しやすい。
    
- `poll` は、計算がいま進められるかを確かめる発想である。
    
- `Future` 自体は実行基盤ではなく、executor や runtime が計算の進行を支える。
    
- `await`、`spawn`、`join` は同じ非同期文脈に出てくるが、待機・起動・合流という別々の責務を持つ。
    

次章では、task・runtime・Event Loop はどこでつながるのかを扱います。  
値、計算、仕事、実行基盤という見取り図をさらにそろえ、非同期処理の全体像を一枚の地図として見られるようにしていきます。
