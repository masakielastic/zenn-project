---
title: "この本の読み方 ― 正解ではなく分け方を学ぶ"
---

**初学者**  
非同期処理って、使っているうちに、わかった気になりやすいですよね。  
`Promise` も `async` / `await` も毎日書いているのに、いざ「Promise って何ですか」と聞かれると、急に説明しづらくなります。

**解説者**  
それは自然なことです。  
非同期処理は、日常的には「動けば使える」からです。`fetch()` を呼んで、`await` して、値を受け取る。その流れが成立してしまうので、途中で何が分かれているのかを意識しなくても前に進めます。

**実装者**  
しかも JavaScript は、かなり親切です。  
ブラウザでも Node.js でも、最初から `Promise` と `await` が使えて、イベントループも普段は見えません。だから、動くコードを書く段階では困りにくいんです。

**初学者**  
でも、少し踏み込むと急に苦しくなります。  
`Promise` と `Future` は何が違うのかとか、`task` って何なのかとか、`Event Loop` は誰の仕事なのかとか。言葉が増えた瞬間に、全部同じものに見えてきます。

**解説者**  
この本は、そこを整理するための本です。  
ただし、最初から「これが唯一の正解です」と言い切る本ではありません。  
この本で重視するのは、非同期処理のAPIを暗記することではなく、**責務を分けて見ること**です。

Promise / Future / Task / runtime / Event Loop の違い、さらに Stream / Socket / Event Loop を同じ種類のものとして混同しない見方を育てること。  
それが本書の狙いです。

**初学者**  
つまり、この本は「正しい定義を一発で覚える本」ではないんですね。

**解説者**  
はい。  
もっと正確に言えば、**ずれない読み方を身につける本**です。  
この章でも、その読み方そのものを先に確認します。

## この章で最初に確認したいこと

**初学者**  
「責務を分ける」と言われても、まだ少し抽象的です。  
非同期処理って、結局は「あとで終わる処理」でしょう、くらいの理解でも普段は困らない気がします。

**解説者**  
普段はそれでも動きます。  
でも、その理解だけだと、少し複雑な設計に入ったところで破綻します。  
本書では、まず「何を同じものとして見てはいけないか」を分けながら読みます。

たとえば、次のようなものを区別せずに扱うと、混乱が始まります。

### 整理

- **値**  
    今あるデータ、または後で受け取る結果として見るもの
    
- **計算**  
    その結果に至る途中の進行として見るもの
    
- **実行単位**  
    スケジューリングされる仕事の単位として見るもの
    
- **実行基盤**  
    その仕事を進める土台として見るもの
    
- **責任スコープ**  
    どこまでまとめて待つか、止めるか、失敗を引き受けるかという範囲
    

**解説者**  
ここでは、まだ厳密定義を完成させる必要はありません。  
まずは、混同しやすい概念を分けるための仮の地図として置いています。

**実装者**  
JavaScript では、これらが一つの書き味に圧縮されがちです。  
`await` は便利ですが、その便利さのせいで、「何を待っているのか」「誰が進めているのか」「その失敗を誰が引き受けるのか」が見えにくくなります。

**初学者**  
なるほど。  
「動くコード」と「責務が整理された説明」は別なんですね。

**解説者**  
その通りです。  
本書は、非同期処理を **「await の書き方」ではなく「責務と実行モデルの地図」として理解する** ことを目指します。  
だから読むときも、「この用語の正解は何か」だけでなく、「いま何をどの責務として見ているか」を確かめながら進めてください。

## なぜ「わかった気」になりやすいのか

**初学者**  
でも、どうして非同期処理だけ、こんなに「わかった気」になりやすいんでしょう。

**解説者**  
理由は二つあります。  
一つは、見えているAPIがきれいすぎること。  
もう一つは、背後の役割分担が隠されていることです。

**実装者**  
たとえば `await fetch(...)` という一行を見ると、全部が一つの仕組みに見えます。  
でも実際には、I/O の発行、待機、再開、結果の受け渡し、失敗の伝播といった別々の仕事が重なっています。

**解説者**  
ここで大事なのは、「一行に見える」ことと「責務が一つである」ことは別だという点です。  
見た目が単純でも、中では分業が起きています。  
この分業を言語化できないと、他言語に移ったときに一気に迷います。

**初学者**  
たしかに、Rust や Python の async を見ると、急に部品が増えた感じがします。  
JavaScript では自然だったものが、別の言語では名前を分けて出してくる感じです。

**解説者**  
本書は、まさにそこを橋渡しします。  
JavaScript ユーザーがすでに持っている感覚を否定するのではなく、それを分解して読み替えることで、他言語でも通用する地図に変えていきます。

**実装者**  
そのときに大事なのは、`await` しているコード自身が全部を進めているわけではない、と最初に意識することです。  
背後には、進行を担う仕組みがあります。  
この本では、その「誰が何を進めるのか」を少しずつ見える形にしていきます。

**解説者**  
だからこの本の読み方も変わります。  
「便利な一行の意味をそのまま信じる」のではなく、  
「その一行の裏で、いくつの役割が分かれているか」を見る。  
それが、本書の基本姿勢です。

## この本の進め方 ― 3つの仮説で読む

**初学者**  
「地図」と言われると少し安心します。  
でも、読み方にもコツがありそうですね。

**解説者**  
あります。  
この本では、対話を三つの仮説で動かします。  
最初から完成した答えを出すのではなく、論点を切り、見方を選び、暫定的な結論を修正していく進め方です。  
これは本書全体の基本方針でもあります。

### 整理

- **論点仮説**  
    いま何を問題として切り出すか
    
- **アプローチ仮説**  
    どこから見ると理解しやすいか
    
- **答え仮説**  
    現時点ではこう整理できそうだ、という暫定結論
    

**実装者**  
実装の話をするときも同じです。  
いきなり「答え」だけ出すと、なぜその見方を採ったのかが見えなくなります。  
非同期処理は特に、見方を間違えると用語が全部ずれてしまうので、まず論点を切ることが大事です。

**初学者**  
つまり、「Promise はこうです」と覚える前に、  
「いま Promise を値として見ているのか、計算として見ているのか」を確かめる必要があるんですね。

**解説者**  
その理解でよいです。  
本書では、用語を一発で固定するより先に、**いまどの棚に置いて読んでいるか**を確認します。  
だから読者も、「定義を覚えたかどうか」だけでなく、「論点を取り違えていないか」を見ながら読んでください。

## なぜ対話形式なのか

**初学者**  
普通の解説書ではなく、対話形式なのもそのためですか。

**解説者**  
その通りです。  
非同期処理の理解では、「誤解してから修正する」流れそのものに価値があります。  
最初からきれいに整理された説明だけだと、読者がどこで混乱しやすいのかが見えにくくなります。

**実装者**  
実際、現場の混乱はこうです。  
「Promise があるなら task も同じでしょう」  
「Stream が読めるなら Event Loop と一体でしょう」  
「await できるなら runtime は意識しなくてよいでしょう」  
こういう早い結論は、全部それなりに自然です。

**初学者**  
たしかに、かなり言いそうです。

**解説者**  
だから本書では、誤解を単なるミスとして扱いません。  
誤解は、どの概念が混線しやすいかを示す手がかりです。  
対話形式にしているのは、その混線と修正の流れを、読者が追えるようにするためです。

**実装者**  
つまり、対話は雰囲気づくりではなく、読み方の補助なんです。  
どこで早い答え仮説が出て、どこで論点を切り直し、どこで実装の現実に接続するのか。  
その順番が見えるようになります。

**解説者**  
ですから、本書の対話は「会話を楽しむ」ためだけではありません。  
読者自身が、誤解からどう立て直すかを学ぶための構成です。

## 3人の登場人物は何を担うのか

**初学者**  
ここで登場人物の役割を、はっきりさせておきたいです。

**解説者**  
では先に整理します。

### 整理

- **初学者**  
    疑問、混乱、誤解を出す。読者のつまずきを代弁する。
    
- **解説者**  
    構造を説明し、論点を切り分け、用語を定義する。
    
- **実装者**  
    挙動や設計上の破綻点を補足し、抽象説明を現実の動きにつなぐ。
    

**実装者**  
大事なのは、役を奪わないことです。  
初学者が急に正解を整理し始めない。  
解説者が延々と独演しない。  
実装者が抽象理論の主役にならない。  
この役割分担が崩れると、対話形式の意味がなくなります。

**初学者**  
なるほど。  
読者の頭の中で起きることを、そのまま役割分担にしている感じですね。

**解説者**  
そうです。  
この本は「答えを読む本」であると同時に、「どう考えればその答えにたどり着くかを学ぶ本」でもあります。  
だから読むときも、「誰が今どの仕事をしているか」を意識すると、対話の使い方が見えやすくなります。

## この本で最初に手放したい期待

**初学者**  
読者としては、少し身構えるところがあります。  
「この本を読めば Promise と Future の正解が一発でわかる」みたいな期待は、持たないほうがいいですか。

**解説者**  
その期待は、少し手放したほうがよいです。  
より正確には、「一発で終わる正解」よりも「ずれない分け方」を手に入れる本だと思ってください。

**実装者**  
非同期処理では、名前が似ていても責務が違うものが多いです。  
逆に、名前が違っていても、見ている責務がかなり近いものもあります。  
だから、本当に必要なのは単語帳ではなく、比較できる地図なんです。

**解説者**  
この本の後半では、ブラウザ、Node.js、Rust、Python を同じ責務マップの上に並べます。  
そのためにも、第1章ではまず「正解」より「分け方」を学ぶ、という読み方に切り替えておく必要があります。

**初学者**  
つまり、読む側も少し姿勢を変える必要があるんですね。  
「答えをもらう」より、「どう切り分けるかを学ぶ」ほうに寄せる。

**解説者**  
その通りです。  
この章でいちばん先に身につけてほしいのは、まさにその読み方です。

### この章の核心

この本は、非同期処理の便利な書き方を覚えるための本ではありません。  
値、計算、実行単位、実行基盤、責任スコープを分けて見るための本です。  
そのために、対話と三つの仮説を使って、誤解を修正しながら理解を組み立てていきます。

## まとめ

- 非同期処理は、`await` が書けるだけでは理解したことにならない。
    
- 本書の目的は、Promise / Future / Task / runtime / Event Loop などを、責務で区別して読めるようになることにある。
    
- この章で出した「値」「計算」「実行単位」「実行基盤」「責任スコープ」は、まず混同を避けるための仮の地図である。
    
- 本書では、論点仮説・アプローチ仮説・答え仮説の三つを使って、正解を急がずに読み進める。
    
- この本の読み方は、「唯一の正解を覚える」ではなく「ずれない分け方を身につける」である。
    

次章では、JavaScript ユーザーがすでに何を知っていて、どこで誤解しやすいのかを整理します。  
Promise や `await` の便利さが、どこで理解の近道になり、どこで混線の原因になるのかを見ていきます。
