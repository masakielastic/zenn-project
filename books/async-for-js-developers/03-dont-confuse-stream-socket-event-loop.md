---
title: "Stream・Socket・Fiber・Event Loop を同じものだと思わない"
---

**初学者**  
ここから急にややこしくなってきました。  
`Stream` も `Socket` も `Task` も `Event Loop` も、非同期処理の話で一緒に出てくるじゃないですか。  
だから、なんとなく「同じグループのもの」だと思っていました。

**解説者**  
それが、この章でほどきたい誤解です。  
たしかに、これらは同じ文脈で登場しやすいです。  
でも、同じ場面で使われることと、同じ種類のものだということは別です。

**実装者**  
JavaScript では特に混ざって見えやすいです。  
Node.js では `stream` や `net.Socket` が出てきますし、`async` / `await` を使うと処理の流れも一続きに見えます。  
そのせいで、データの通り道、I/O の接続口、進行中の仕事、実行の交通整理が、全部同じ棚に入ってしまいやすいんです。

**初学者**  
まさにそれです。  
全部「非同期のための道具」くらいに思っていました。

**解説者**  
この章では、そこを責務で分けます。  
中心命題は一つです。  
**一緒に登場するからといって、同じ種類のものとは限らない**。  
非同期処理を理解するには、まずこの分類感覚が必要です。

## なぜ同じものに見えやすいのか

**初学者**  
でも、どうしてそんなに混ざって見えるんでしょう。

**解説者**  
理由は単純です。  
どれも「すぐには終わらない処理」に関わるからです。  
ネットワーク通信をすると `Socket` や `Stream` が出てきます。  
複数の処理を並べて進める話になると、進行中の仕事の単位が気になってきます。  
そして、それらをどう進めるかを考えると `Event Loop` が出てきます。

**実装者**  
つまり、同じ現場に集まってくるんです。  
でも、同じ現場にいるからといって、同じ職種ではありません。

**初学者**  
工事現場にいる人が全員同じ仕事をしているわけではない、みたいな感じですか。

**解説者**  
その通りです。  
この章では、その職種の違いを言葉にしていきます。

## まずは分類の軸を置く

**初学者**  
いきなり個別の単語を説明するんじゃないんですね。

**解説者**  
はい。  
先に分類の軸を置いたほうが、個々の API 名に引きずられにくいからです。  
少なくとも、次のように分けて考えると整理しやすくなります。

### 整理

- **値**  
    読み取られるデータそのもの
    
- **ハンドル**  
    外部資源に対して操作を行うための窓口
    
- **実行単位**  
    進めたり、待ったり、再開されたりする仕事の単位
    
- **実行基盤**  
    複数の仕事を監視し、進行を調停する土台
    

**実装者**  
この章では、`Socket` と `Stream` をハンドルとして、`Task` を実行単位として、`Event Loop` を実行基盤として見ていきます。

**初学者**  
ここで少し驚きました。  
`Stream` と `Socket` はデータのことじゃないんですね。

**解説者**  
そこが最初の重要点です。  
データが流れるからといって、`Stream` 自体がデータではありません。  
`Socket` も通信内容そのものではありません。  
どちらも、外部のものに対して読み書きを行うための接点です。

## Socket は何者か

**初学者**  
では、`Socket` からお願いします。  
名前は知っていますが、かなりふわっとしています。

**解説者**  
`Socket` は、ネットワーク越しの相手とつながるための操作口です。  
大事なのは、相手そのものではなく、**相手に対して送受信するための窓口**だという点です。

**実装者**  
コードの感覚で言えば、`Socket` は「この接続に対して読み書きする能力を持ったもの」です。  
接続の状態を持ち、送る、受け取る、閉じるといった操作の入り口になる。

**初学者**  
ということは、`Socket` は自分で勝手に全体を進めるわけではないんですね。

**解説者**  
その通りです。  
`Socket` は操作を持っていますが、処理全体の司令塔ではありません。  
何を送るか、いつ待つか、失敗したらどうするかを決めるのは別の場所です。

## Stream は何者か

**初学者**  
では `Stream` はどう違うんでしょう。  
Node.js の `stream` はよく見かけますが、`Socket` とかなり近く見えます。

**解説者**  
近い関係ではあります。  
ただし、注目しているものが少し違います。  
`Socket` は接続という外部資源への窓口として見やすい。  
`Stream` は、読み書きの流れとして見やすい。

**実装者**  
Node.js の `Readable` や `Writable` を思い浮かべるとわかりやすいです。  
それらは仕事全体の管理者ではなく、データを受け渡すためのインターフェースです。  
読み込みや書き込みを扱いやすくするための形を提供している。

**初学者**  
つまり、`Stream` は「流れるデータそのもの」ではなく、「流れとして扱うための窓口やモデル」なんですね。

**解説者**  
その理解でよいです。  
ここでも重要なのは、`Stream` もまた全体の進行役ではないということです。  
自分で複数の仕事を調停したり、アプリケーション全体の進め方を決めたりはしません。

**実装者**  
Node.js では `net.Socket` が stream 系のインターフェースも持つので、接続の窓口とデータ流の窓口が一つのオブジェクトに見えやすいです。  
だから `Socket` と `Stream` が余計に混ざるんです。  
でも概念上は、  
`Socket` はネットワーク接続のハンドル、  
`Stream` は読み書きの流れを扱うための抽象化、  
と分けておいたほうが理解しやすいです。

**初学者**  
名前だけ追うと同じものに見えるけど、注目点が違うんですね。

**解説者**  
そうです。  
同じオブジェクトに複数の顔が乗ることはあります。  
でも、責務まで同じになるわけではありません。

**実装者**  
`Stream` は便利なので、つい何でも背負わせたくなります。  
読み込み、書き込み、バッファ管理、エラー処理、再試行、タイムアウト、上位プロトコルの状態管理まで入れたくなる。  
でもそれをやると、すぐに責務が膨らみます。

**初学者**  
「動く」と「責任を持つ」の違いが、ここでも効いてくるんですね。  
`Stream` にいろいろ載せれば動くかもしれないけど、それで責務が正しいとは限らない。

**解説者**  
その通りです。

## Task は何者か

**初学者**  
次は `Task` ですね。  
これは `Stream` や `Socket` よりかなり違う感じがします。

**解説者**  
はい。  
ここでいう `Task` は、外部資源の窓口ではありません。  
また、HTML のイベントループ文脈でいう task queue の `task` そのものを指しているわけでもありません。  
この章では、**非同期処理における進行中の仕事の単位**という、少し広い意味で使います。

**初学者**  
先にそこを分けておくんですね。

**解説者**  
はい。  
JavaScript では `task` という言葉が別の意味でも使われるので、ここを曖昧にすると後で混乱しやすいからです。

**実装者**  
たとえば、ある `async function` を走らせるとします。  
その処理は途中で待つかもしれないし、あとで再開されるかもしれない。  
そういう「進行する仕事」を一つの単位として見たものが、この章でいう `Task` です。

**初学者**  
すると、`Task` は「何かを読むもの」ではなく、「進められる処理そのもの」なんですね。

**解説者**  
そうです。  
`Socket` や `Stream` が外部資源への接点だとすると、`Task` はプログラム側の仕事のまとまりです。

**初学者**  
`async function` で書いている処理の流れに近いですか。

**解説者**  
かなり近いです。  
ただし、この章で大事なのは API の細部ではありません。  
重要なのは、`Task` は I/O オブジェクトではなく、仕事の単位だということです。

**実装者**  
I/O を待つ場面では `Socket` も `Stream` も `Task` も関わります。  
でも、  
何に対して操作しているのか、  
何を進めているのか、  
どの単位で待ちや再開が起こるのか、  
は別の話です。

## Event Loop は何者か

**初学者**  
では `Event Loop` はどう位置づければいいでしょう。  
これが一番なんでも屋に見えます。

**解説者**  
`Event Loop` は、少なくとも `Stream` や `Socket` や `Task` と同列には置かないほうがよいです。  
`Event Loop` は、**複数の仕事の進行を調停する実行基盤**として見るのが基本です。

**実装者**  
くだけて言えば交通整理です。  
どのイベントが来たか、どの仕事を進めるか、今は何を処理するかを回していく。  
ただし、ここでも注意があります。  
`Event Loop` が全部を一人でやる超主体だと思うと、また混乱します。

**初学者**  
またそこで、見え方と責務を分ける必要があるんですね。

**解説者**  
そうです。  
`Event Loop` は監視、通知、キュー処理、再開のきっかけ作りに関わります。  
でも、読み書きの窓口そのものではありませんし、個々の処理内容そのものでもありません。

**初学者**  
つまり、`Event Loop` は `Socket` ではないし、`Stream` でもないし、`Task` でもない。

**解説者**  
はい。  
もっと言えば、`Event Loop` はそれらを同じ棚で比較する対象ではなく、  
それらが関わる仕事の流れをどう前に進めるかという、より上位の土台です。

**実装者**  
ただし、ここで一つだけ先に言っておきます。  
「待てるようになったことを知ること」と、「次の処理を実際に進めること」は、頭の中ではいったん分けて考えたほうが整理しやすいです。  
この点は次章で詳しく見ます。

## 4つを並べて見直す

**初学者**  
ここまで来ると、だいぶ違いが見えてきました。  
でも一度並べて整理したいです。

**解説者**  
では、この章の中心をそのまま言い換えます。

### 整理

- **Socket**  
    ネットワーク接続という外部資源に対するハンドル
    
- **Stream**  
    読み書きの流れを扱うためのハンドルや抽象化
    
- **Task**  
    進行中の仕事を表す実行単位
    
- **Event Loop**  
    複数の仕事を監視・調停して進める実行基盤
    

**実装者**  
この整理の利点は、責務がずれにくくなることです。  
たとえば `Stream` にスケジューラの責務を載せない。  
`Task` に I/O ハンドルの責務を載せない。  
`Event Loop` を単なるデータ通路だと思わない。  
この区別だけでも、設計はかなり安定します。

**初学者**  
今までの自分は、  
`Stream` はなんとなく非同期そのもの、  
`Event Loop` はなんとなく裏側の魔法、  
くらいの理解でした。  
かなり雑でしたね。

**解説者**  
雑だったというより、日常的に使うぶんにはそれでも足りていたんです。  
この本は、その日常理解を設計に耐える形へ引き上げるための本です。

## なぜこの分類が他言語でも効くのか

**初学者**  
この分類は、JavaScript 以外でも効くんでしょうか。

**解説者**  
そこが重要です。  
この分類は、特定の言語の API 名に依存していません。  
ハンドル、実行単位、実行基盤という分け方は、Rust や Python を読むときにも役立ちます。

**実装者**  
言語が変わると、名前や見え方は変わります。  
でも、外部資源への窓口なのか、仕事の単位なのか、全体を回す基盤なのか、という問いは共通です。

**初学者**  
単語を暗記するというより、「これはどの職種か」を判定する感覚を作るわけですね。

**解説者**  
その通りです。  
そして、この感覚がないまま async の設計に入ると、いろいろな責務が一つのオブジェクトに集まり始めます。  
それが次の混乱の種になります。

### この章の核心

`Stream`、`Socket`、`Task`、`Event Loop` は、同じ場面で登場しても同じ種類のものではありません。  
`Stream` と `Socket` は外部資源へのハンドル、`Task` は実行単位、`Event Loop` は実行基盤として分けて見ることで、非同期処理の責務が見えやすくなります。

## まとめ

- 一緒に登場するものを同じ種類だと思うと、非同期処理の設計はすぐ混線する。
    
- `Socket` はネットワーク接続に対するハンドルであり、処理全体の司令塔ではない。
    
- `Stream` は読み書きの流れを扱うためのハンドルや抽象化であり、実行基盤ではない。
    
- `Task` は進行中の仕事を表す実行単位であり、I/O オブジェクトではない。
    
- `Event Loop` は複数の仕事を調停する実行基盤であり、個々の接続口やデータ通路とは別物である。
    

次章では、`Event Loop` が実際には何を回しているのかを見ていきます。  
特に、待てる状態を知ることと、次の処理を進めることの違いを整理します。
