---
title="task・runtime・Event Loop はどこでつながるのか"
---

**初学者**  
前の章で、`Promise` は結果寄り、`Future` は計算寄り、という整理が出てきました。  
でも、まだ頭の中で少し散らばっています。  
`task`、`runtime`、`Event Loop` がどこでつながるのかが、まだ一枚の図になっていません。

**解説者**  
この章は、そこをつなぐ章です。  
ここまでで、値、計算、ハンドル、実行基盤という部品は少しずつ分かれてきました。  
でも、部品が分かるだけでは全体像にはなりません。  
この章では、それらが **どう関係しながら一つの非同期処理系を形づくるのか** を見ます。

**実装者**  
JavaScript ユーザーがここで混乱しやすいのは自然です。  
日常では `await` すれば処理が続いていくので、値と計算と仕事の単位と、それを支える基盤が、一つの書き味に圧縮されています。

**初学者**  
まさにそうです。  
だから `task` も `runtime` も `Event Loop` も、「裏側の何か」くらいの感じで一緒に見えてしまいます。

**解説者**  
この章の中心命題は一つです。  
**`task`、`runtime`、`Event Loop` は同じものではないが、互いに接続して非同期処理を成立させている。**  
その接続の仕方を、値・計算・仕事・基盤という順で整えていきます。

## まず「値」「計算」「仕事」「基盤」を並べる

**初学者**  
最初に全体の棚を作るわけですね。

**解説者**  
はい。  
この棚がないと、用語が増えるたびにまた混ざります。  
まずは四つに分けます。

### 整理

- **値**  
    いま得られているデータ、または後で得られる結果
    
- **計算**  
    まだ完了しておらず、進める必要がある処理
    
- **仕事**  
    実行基盤の上でスケジュールされる単位
    
- **基盤**  
    複数の仕事を監視・調停・再開する土台
    

**実装者**  
この章では、  
値に `Promise` 的な見方、  
計算に `Future` 的な見方、  
仕事に `task`、  
基盤に `runtime` や `Event Loop`  
を対応させて考えると整理しやすいです。

**初学者**  
ここまで来ると、かなり図にしやすそうです。  
`Promise = 値`、`Future = 計算`、`task = 仕事`、`runtime/Event Loop = 基盤` という感じですね。

**解説者**  
入口としてはその理解でよいです。  
もちろん言語やライブラリによって重なり方は違います。  
でも責務の地図としては、この分け方がかなり効きます。

## task は「動いている仕事の単位」として見る

**初学者**  
まず `task` から確認したいです。  
前の章では「基盤の上に載せられた仕事の単位」と言っていましたが、もう少し実感を持ちたいです。

**解説者**  
`task` は、単なる抽象的な計算ではなく、**実際に進行対象になっている仕事** として見るとわかりやすいです。  
つまり「こういう計算があります」だけではなく、「この計算をこの基盤で進めます」という段階に入ったものです。

**実装者**  
ここで重要なのは、`task` は `Future` と同一ではない、という点です。  
`Future` は計算の形として見られる。  
`task` は、その計算が runtime に載り、進行・停止・再開・完了の管理対象になったものとして見られる。

**初学者**  
すると `task` は、計算を実務の世界に持ち込んだもの、みたいな感じですね。

**解説者**  
その表現でかなり近いです。  
第6章の言い方で言えば、`Future` が「こういう計算がある」という見え方なら、`task` は「その計算をこの基盤でいま扱っている」という見え方に近いです。  
`task` になると、単に結果を待つだけでなく、  
いつ始まるか、  
いつ終わるか、  
途中で失敗したらどう扱うか、  
親の仕事とどうつながるか、  
といった管理の問いが前に出てきます。

## runtime は何を支えているのか

**初学者**  
では `runtime` は、その `task` を動かす土台なんですね。

**解説者**  
はい。  
この本では、`runtime` を **仕事が動ける全体の土台** として捉えます。  
計算があり、仕事として管理され、再開や待機が起きる。  
そうした一連の流れを成立させる側です。

**実装者**  
ここで `runtime` は、単なる一つの関数や一つのオブジェクトではなく、もっと広い役割として見ると理解しやすいです。  
仕事のスケジュール、待機、再開、場合によってはタイマーやI/Oとの接続など、いろいろな要素を含む土台です。

**初学者**  
`runtime` という言葉は広いですね。  
少しぼんやりしませんか。

**解説者**  
ぼんやりしやすいからこそ、対比が必要です。  
`runtime` は `task` ではありません。  
`task` は土台の上で動く仕事です。  
`runtime` は、その仕事が動けるようにする側です。

### 整理

- `task` は、動いている仕事の単位
    
- `runtime` は、その仕事を動かすための土台
    
- `task` が主役のように見える場面でも、背後には `runtime` が必要
    
- したがって、仕事と基盤は分けて考える必要がある
    

## Event Loop は runtime の中でどう見えるか

**初学者**  
では `Event Loop` は `runtime` と同じなんでしょうか。  
ここがまだ少し引っかかります。

**解説者**  
完全に同じとは考えないほうが整理しやすいです。  
`Event Loop` は、runtime の中でも **進行を調停する中核的な仕組み** として見ると理解しやすいです。

**実装者**  
前の章でも出てきた通り、`Event Loop` は交通整理に近いです。  
何が進行可能になったか。  
どの仕事を再開できるか。  
どのイベントを処理するか。  
そうした判断の流れに関わる。

**初学者**  
ということは、runtime がもっと広い土台で、その中に Event Loop 的な部分がある、という理解ですか。

**解説者**  
はい。  
この本ではその見方で進めます。  
runtime を、仕事が成り立つ全体の土台として見る。  
Event Loop を、その中で監視・通知・調停を担う重要部分として見る。  
そうすると両者を無理なくつなげられます。

**初学者**  
ここでようやく位置関係がわかってきました。  
Event Loop が全部ではないけれど、かなり重要な中心部なんですね。

**解説者**  
その通りです。  
つまり Event Loop は、runtime 全体の別名ではなく、runtime の中で「いまどれを進めるか」に強く関わる部分として読むほうが整理しやすいです。

## task はどうやって runtime とつながるのか

**初学者**  
ここで一番知りたいのは、`task` がどうやって `runtime` とつながるのかです。

**解説者**  
よい問いです。  
一言で言えば、**task は runtime によって進行可能な単位として管理される**、です。

**実装者**  
計算があるだけでは足りません。  
その計算が、いま進められるのか、少し待つべきなのか、何かのイベントを待っているのかを見ながら、適切なタイミングで再開される必要があります。  
その管理の中に task が入ります。

**初学者**  
つまり `task` は、runtime にとっての管理対象なんですね。

**解説者**  
その通りです。  
`task` をただの「処理の塊」として見るだけでは弱い。  
runtime の上で、待機や再開や完了の単位として扱われる仕事だと見るほうが、設計に効きます。

### 整理

- 計算が存在するだけでは、自動では進まない
    
- 仕事として管理される単位が `task`
    
- その `task` を進行させる土台が `runtime`
    
- 進行可能性の観測と調停に `Event Loop` 的な仕組みが関わる
    

## callback と continuation は何が違うのか

**初学者**  
ここで少し別の疑問です。  
`callback` と `continuation` は同じですか。  
どちらも「続き」に見えます。

**解説者**  
似た場面で使われますが、同じものとして潰さないほうがよいです。  
この章では細かな理論に入りすぎず、役割の違いだけ押さえます。

**実装者**  
`callback` は、何かが起きたときに呼ばれる関数として見やすいです。  
一方 `continuation` は、処理の続きそのものという見方が前に出ます。  
つまり、「イベント発生時に呼ぶもの」と「中断した処理の再開地点」は、近いけれど同一ではありません。

**初学者**  
なるほど。  
イベントで関数を呼ぶことと、途中で止まっていた処理の続きを持っていることは、たしかに違いますね。

**解説者**  
ここが大事です。  
`Event Loop` は callback を起点に説明されることがあります。  
でも task や async/await を考えるときには、continuation、つまり「再開される続き」の感覚も必要になります。

**初学者**  
だからイベント駆動の説明だけだと、task の理解には少し足りないんですね。

**解説者**  
その通りです。

### 整理

- **callback**  
    何かの出来事を受けて呼ばれる関数として見やすい
    
- **continuation**  
    中断した処理の続きとして見やすい
    
- 非同期処理では両者が近い場面で現れるが、責務の見え方は異なる
    
- task と runtime を考えるときは、continuation の感覚が重要になる
    

## JavaScript では、なぜこのつながりが見えにくいのか

**初学者**  
ここまでの整理はかなり納得感があります。  
でも、なぜ JavaScript だとこんなにつながりが見えにくいんでしょう。

**解説者**  
理由は、抽象化がうまく機能しているからです。  
JavaScript では、結果の受け取り、待機、イベント処理、継続の再開が、かなりきれいな書き味に圧縮されています。

**実装者**  
たとえば `async` 関数を書いて `await` するだけで、  
「将来の値」  
「途中で止まる処理」  
「あとで再開される続き」  
「それを支える実行基盤」  
の多くが裏側に隠れます。  
そのため、動くコードは書きやすいけれど、責務の地図は見えにくい。

**初学者**  
便利さの代償として、部品の境界が見えにくくなるわけですね。

**解説者**  
はい。  
ただし、それ自体は悪いことではありません。  
日常ではその抽象化が大きな助けになります。  
この本でやっているのは、その抽象化を一度ほどいて、設計に耐える地図へ戻している作業です。

## Monitor・Scheduler・Dispatcher として見る

**初学者**  
ここまでの話を、もう少し役割語で整理できますか。  
監視とか調停とか、いくつか言葉が出てきたので。

**解説者**  
できます。  
ここでは厳密な用語論をするというより、役割の違いを見失わないためのラベルとして使いましょう。  
ここまでの説明を、役割語でさらに粗く切り分けると見通しがよくなります。

### 整理

- **Monitor**  
    何が進行可能になったかを観測する役割
    
- **Scheduler**  
    どの仕事をいつ進めるかを決める役割
    
- **Dispatcher**  
    実際に仕事の再開や呼び出しにつなぐ役割
    

**実装者**  
現実のランタイムでは、これらがぴったり別部品になっていないこともあります。  
でも、責務として分けて考えると非常に見通しがよくなります。  
Event Loop をただの一語で終わらせず、その中に monitor、scheduler、dispatcher 的な働きがあると見れば、runtime との関係も説明しやすい。

**初学者**  
ここでかなり一枚の地図になってきました。  
task が仕事の単位、runtime が土台、Event Loop がその中の調停の中心、さらに monitor や scheduler や dispatcher の役割がある。  
そんな感じですね。

**解説者**  
はい。  
この地図を持てるようになると、非同期処理を「await の便利さ」だけでなく、実行モデルとして読めるようになります。

## ここまでの全体像を一度つなぐ

**初学者**  
最後に、ここまでの部品を一度つないでほしいです。  
文章で追える一枚絵がほしいです。

**解説者**  
では、順に並べます。  
ある計算がある。  
その計算は結果に向かって進む。  
結果側から見ると将来値として見え、計算側から見ると `Future` のように見える。  
その計算が実行基盤の上で管理対象になると `task` として見える。  
その `task` を進める土台が `runtime` であり、進行可能性を観測し、再開を調停する中核として `Event Loop` 的な仕組みが働く。  
そして呼び出し側からは、その結果に `await` で接続する。

**実装者**  
つまり、呼び出し側が見ているのはたいてい結果の受け取りです。  
でもその背後には、計算、仕事、基盤、監視、調停、再開がある。  
この多層構造を一語で潰さないことが大事です。

**初学者**  
ようやく「どこでつながるのか」が見えました。  
同じものではないけれど、ちゃんと線で結べるんですね。

### この章の核心

`task`、`runtime`、`Event Loop` は同じものではありません。  
`task` は基盤の上で管理される仕事の単位であり、`runtime` はその仕事が動ける土台であり、`Event Loop` はその中で進行可能性の観測と調停を担う中核です。  
この関係を、値・計算・仕事・基盤の順で見ると、非同期処理の全体像がつながります。

## まとめ

- 非同期処理は、値・計算・仕事・基盤に分けて見ると整理しやすい。
    
- `task` は、実行基盤の上で管理される仕事の単位である。
    
- `runtime` は、その仕事が動ける全体の土台である。
    
- `Event Loop` は runtime の中で、進行可能性の観測と調停を担う重要な仕組みである。
    
- callback と continuation は近いが同じではなく、task の理解には continuation の感覚が重要である。
    

次章では、ここまでの地図を使って、なぜ「動くコード」でも設計は破綻するのかを見ていきます。  
野良タスク、例外の取りこぼし、キャンセルの不整合といった問題を、責務の混線として読み解いていきます。
