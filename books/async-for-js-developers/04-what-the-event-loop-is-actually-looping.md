---
title: "イベントループは何を回しているのか"
---

**初学者**  
前の章で、`Event Loop` は `Stream` や `Socket` と同じ種類のものではない、という話が出てきました。  
でも正直、まだ少し曖昧です。  
イベントループって、結局は何をしているんですか。何を「回している」んでしょう。

**解説者**  
とても大事な問いです。  
この章の中心はそこです。  
`Event Loop` という言葉はよく出てきますが、言葉の印象だけで理解すると、「裏で全部を動かしてくれる謎の装置」になりやすい。  
でも実際には、もう少し分けて考えたほうがわかりやすいです。

**実装者**  
特に JavaScript ユーザーは、`Promise` や `await` の書き味が自然なので、イベントループの存在をあまり意識しなくても日常のコードが書けます。  
そのぶん、「イベントループが何を担当していて、何を担当していないのか」が見えにくくなりやすいです。

**初学者**  
たしかに、普段は「イベントループがなんとかしてくれるもの」くらいに思っています。

**解説者**  
その感覚は自然です。  
ただ、この本ではそこから一歩進んで、`Event Loop` を **仕事の交通整理をする実行基盤**として見ていきます。

## 「回す」という言い方が誤解を生みやすい

**初学者**  
まずそこからなんですね。

**解説者**  
はい。  
「イベントループは何を回しているのか」という問いは便利ですが、同時に少し危険でもあります。  
なぜなら、「回す」という言い方が、まるでイベントループが自分で全部の処理を直接実行しているように聞こえるからです。

**実装者**  
実際には、そんな単純ではありません。  
イベントが発生する。  
待っていた仕事が再開可能になる。  
キューに入っている仕事が取り出される。  
どの続きが進められるかが決まる。  
こうした流れの調停に関わるのがイベントループです。

**初学者**  
つまり、イベントループは「仕事の中身そのもの」を持っているというより、「何を先に進めるかを整理する側」なんですね。

**解説者**  
その理解でよいです。  
だからこの章では、「何を実行するか」だけでなく、**誰が何を観測し、何をきっかけに再開するのか**も含めて見ていきます。

## まずはイベントキューという考え方

**初学者**  
イベントループの前に、イベントキューの話が必要そうですね。

**解説者**  
はい。  
イベントループを理解するには、まず「待っている仕事がどこかに並ぶ」という感覚が必要です。

**実装者**  
たとえば、ユーザーがボタンを押す。  
タイマーが満了する。  
ネットワークからデータが届く。  
ファイルI/Oが終わる。  
こうした「進めてよいきっかけ」が発生すると、その情報をもとに、対応する仕事が後で進められます。

**初学者**  
その「後で進められる候補」が並ぶ場所がイベントキュー、みたいな理解でいいですか。

**解説者**  
入口としてはそれでよいです。  
ただし本書では、イベントそのものがそのまま並ぶというより、**イベントを受けて進められる仕事の候補が並ぶ**と考えたほうが整理しやすいです。  
大事なのは、いま目の前で同期的に全部を処理しているわけではなく、**進行の候補が並び、それを順に扱う構造**があることです。

### 整理

- イベントは「何かが起きた」というきっかけ
    
- キューは「進められる候補」が並ぶ場所
    
- イベントループは、その候補を取り出して進行を調停する
    
- したがって、イベントループは単なる反復ではなく、進行管理の仕組みとして見る必要がある
    

**初学者**  
ここで少し見え方が変わってきました。  
イベントループって、イベントそのものではなく、イベントを受けて仕事を前に進めるための仕組みなんですね。

**解説者**  
その通りです。

## イベントループは「交通整理」として見るとわかりやすい

**初学者**  
さっきから「交通整理」という表現が出ていますが、それはかなりしっくりきます。

**解説者**  
この章ではその比喩が有効です。  
ただし最後は正式な言葉に戻します。  
交通整理というのは、車を自分で運転することではありません。  
交差点で、どちらを先に通すか、どこで待たせるか、詰まっていないかを見ることです。

**実装者**  
イベントループもそれに近いです。  
ソケットそのものになるわけではない。  
ストリームそのものになるわけでもない。  
処理の本体そのものになるわけでもない。  
でも、複数の進行候補があるなかで、どれを今進めるかを決める土台になる。

**初学者**  
なるほど。  
前の章の分類で言えば、イベントループはハンドルでも値でもなく、やっぱり実行基盤なんですね。

**解説者**  
はい。  
そして、ここでさらに一段大事なのは、イベントループを「全部を一人でやる主体」として扱わないことです。  
実際には、監視、通知、キュー処理、再開のきっかけ作りなど、いくつかの役割が関わっています。

## 「監視」と「実行」は分けて考える

**初学者**  
ここが少し難しそうです。  
イベントループが全部をやっているのではないなら、何を分ければいいんでしょう。

**解説者**  
まず分けたいのは、**監視** と **実行** です。

**初学者**  
監視と実行。

**解説者**  
はい。  
たとえばソケットが読み取り可能になったかを見ること。  
タイマーが期限に達したかを見ること。  
ある処理が完了したかを知ること。  
これらは「観測」の側面があります。

一方で、その結果を受けてコールバックを呼ぶ、継続を再開する、次の仕事を進める、といった部分は「実行」や「調停」の側面です。

**実装者**  
ここを一つに見てしまうと、イベントループが魔法の黒箱になります。  
でも分けて考えると、  
「何かを監視している部分」  
「進められる仕事を並べる部分」  
「実際に進める部分」  
があると見えてきます。

**初学者**  
たしかに、読み取り可能になったことを知るのと、実際にその続きを動かすのは別の話ですね。

### 整理

- **監視**  
    何が進行可能になったかを観測する
    
- **通知**  
    進められることを他の部分に伝える
    
- **調停**  
    どの仕事をいつ進めるかを決める
    
- **実行**  
    実際に継続やコールバックを進める
    

**解説者**  
この区別は、今後 `runtime` や `task` を考えるときにも効いてきます。  
イベントループを一枚岩に見ないことが大切です。

## readiness モデルとは何か

**初学者**  
ここで readiness モデルが出てくるんですね。

**解説者**  
はい。  
イベントループを理解するうえで、まずは **readiness モデル** から入るとわかりやすいです。  
readiness は、くだけて言えば、**「準備ができたら教えて」** という考え方です。

**実装者**  
ソケットが読めそうです。  
書けそうです。  
タイマーが来ました。  
そういう「今なら進められる」という状態を観測し、それをきっかけに次へ進む。  
この発想が readiness モデルです。

**初学者**  
つまり、処理そのものを代わりにやってくれるのではなく、「今ならやってよい」という合図をくれる感じですか。

**解説者**  
そうです。  
イベントループは、その合図を受け取り、どの仕事を再開するかを調停します。  
JavaScript や Node.js の文脈でイベントループを考えるとき、まずはこの「準備ができたら進める」見方が基本になります。

**初学者**  
この見方だと、イベントループは「完了した仕事の実体」ではなく、「進行可能になった仕事を扱う場所」に近いんですね。

**解説者**  
その通りです。  
そしてここでは、さきほどの整理でいう **監視** と **通知** が前面に出ています。  
何が進められる状態になったかを観測し、それを次の進行へつなぐわけです。

## completion モデルとは何か

**初学者**  
では completion モデルは何が違うんでしょう。

**解説者**  
**completion モデル** は、くだけて言えば、**「これをやっておいて、終わったら教えて」** です。  
つまり、準備ができたことを知らせるよりも、仕事そのものの完了通知に重心があります。

**実装者**  
readiness では、「今読めそうだから、自分で読みなさい」という感じです。  
completion では、「頼んだ仕事が終わったので、結果を受け取りなさい」に近い。  
ここでイベントループやランタイムの見え方が変わってきます。

**初学者**  
ということは、同じ「非同期処理」でも、何を通知してもらうかが違うんですね。  
準備完了なのか、仕事完了なのか。

**解説者**  
その理解でよいです。  
この違いを知らないと、他言語や他ランタイムを見たときに、「なぜこんなに部品の見え方が違うのか」がわからなくなります。

### 整理

- **readiness**  
    準備ができたことを知らせる
    
- **completion**  
    頼んだ仕事が終わったことを知らせる
    
- 監視のしかたと通知のされかたが違うと、調停や再開の見え方も変わる
    
- イベントループやランタイムの設計は、この違いによって見え方がかなり変わる
    
- JavaScript / Node.js の感覚だけで全部を読むと、この差を見落としやすい
    

## Node.js の裏側では何が支えているのか

**初学者**  
Node.js の話だと、libuv という名前もよく出てきます。  
これはイベントループそのものなんでしょうか。

**解説者**  
ここでも、完全に一語で片づけないほうが理解しやすいです。  
Node.js の裏側には、各種の I/O やタイマーなどを扱う基盤があります。  
libuv は、その土台を支える重要な要素の一つです。

**実装者**  
より実務的に言えば、Node.js で「裏側がなんとかしてくれている」部分のかなり大きな一角を担っています。  
ただし、名前を知っただけで責務が全部わかるわけではありません。  
大事なのは、そこにも **監視**、**通知**、**調停** のような役割があると見ることです。

**初学者**  
Node.js のイベントループは、単なる `while` 文のイメージよりずっと大きいですね。

**解説者**  
その通りです。  
だから「イベントループがある」と聞いたときは、単なる繰り返しではなく、**I/O やタイマーやキュー処理を含む進行管理の土台**を想像したほうがよいです。

## Node.js で責務が見えにくい理由

**初学者**  
ここまで聞くと、Node.js ってかなり多くのことを隠してくれているんですね。

**解説者**  
はい。  
それが便利さでもあり、学習上の落とし穴でもあります。

**実装者**  
Node.js では、ソケットやストリームを扱いながら、同時に `Promise` や `await` も使えます。  
さらにイベントループが表ではあまり露出しません。  
すると、I/O ハンドル、継続の再開、キュー処理、実行基盤が、一つの書き味に圧縮されて見えやすいです。

**初学者**  
だから「await すれば終わる」で済ませてしまいやすいんですね。

**解説者**  
そうです。  
もちろん日常のアプリケーションでは、それで十分な場面も多いです。  
でも設計を学ぶなら、  
誰が進行可能性を観測しているのか。  
誰が仕事を並べているのか。  
誰が継続を進めているのか。  
を分けて考えたほうが、後で強くなれます。

**実装者**  
つまり、ここまで見てきた **監視**、**通知**、**調停**、**実行** が、一つの書き味の中に折りたたまれて見えているわけです。  
Node.js は便利ですが、その便利さが責務を見えにくくもします。

## イベントループを「万能装置」にしない

**初学者**  
ここまでの話を聞いて、イベントループを何でも屋だと思わないことが大事だとわかってきました。

**解説者**  
それがこの章の要点です。  
イベントループは重要ですが、万能装置ではありません。  
ハンドルの責務を代わりに持つわけではない。  
個々の仕事の意味そのものを理解するわけでもない。  
失敗処理や寿命管理のすべてを一人で背負うわけでもありません。

**実装者**  
設計が崩れるときは、イベントループという言葉に責務を押し込みすぎていることがよくあります。  
「イベントループがうまくやってくれるでしょう」で済ませると、実際には、タイムアウトは誰が決めるのか、キャンセルは誰が伝えるのか、どこまでを一つの仕事として管理するのかが曖昧になります。

**初学者**  
なるほど。  
イベントループは大事だけど、そこに全部を詰め込むと、逆に何も見えなくなるんですね。

**解説者**  
はい。  
だから本書では、イベントループを「全部を動かす主体」ではなく、**進行可能になった仕事を観測し、並べ、調停し、再開のきっかけを作る実行基盤**として読みます。  
この読み方ができると、後で `Promise` や `task` や `runtime` を見たときにも、責務を押し込みすぎずに済みます。

### この章の核心

イベントループは、何かを「全部実行する謎の装置」ではありません。  
進行可能になった仕事を観測し、並べ、調停し、再開のきっかけを作る実行基盤として見ると、責務が見えやすくなります。  
その理解のためには、監視と実行、readiness と completion を分けて考えることが重要です。

## まとめ

- イベントループは、複数の仕事の進行を調停する実行基盤である。
    
- 「回す」という言い方だけで理解すると、全部を直接実行しているように誤解しやすい。
    
- イベントそのものより、イベントを受けて進められる仕事の候補が並ぶと考えたほうが整理しやすい。
    
- 監視、通知、調停、実行は分けて考えたほうが見通しがよい。
    
- readiness は「準備ができたら教えて」、completion は「終わったら教えて」という違いで捉えるとよい。
    
- Node.js では多くの責務が書き味の中に包まれているため、イベントループを万能装置として読まないことが大切である。
    

次章では、`Promise` は何者で、何をしていないのかを扱います。  
JavaScript ユーザーにとって最も身近な非同期の道具を、値、待機、責任範囲という観点から切り分けていきます。
