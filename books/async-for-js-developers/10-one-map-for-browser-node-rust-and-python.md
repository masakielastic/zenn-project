**初学者**  
ここまでで、かなり地図はできてきました。  
でも、いざ他の言語や実行環境を見ると、また名前に引っぱられそうです。  
ブラウザ、Node.js、Rust、Python って、やっぱりかなり違って見えます。

**解説者**  
違って見えるのは自然です。  
実際、API の形、露出している概念、日常で触れる対象はかなり違います。  
ただ、この章で確認したいのは、**見た目が違っても、責務の棚はある程度そろえて見られる**ということです。

**実装者**  
ここまで本書で分けてきたのは、まさにそのためです。  
値、計算、実行単位、ハンドル、実行基盤、責任スコープ。  
この棚を持っておけば、名前が違っても慌てにくくなります。

**初学者**  
つまり、この章は「翻訳表」を作る章なんですね。

**解説者**  
そうです。  
ただし、単語を一対一に置き換えるだけの表ではありません。  
この章の中心命題は一つです。  
**異なる言語や環境でも、まずは責務の位置で見ると、似ているものと違うものを落ち着いて見分けられる。**

## まずは責務マップを置く

**初学者**  
最初に、棚をもう一度確認したいです。

**解説者**  
では、この本の責務マップをここで明示します。

### 整理

- **値**  
    いま使えるデータ、または後で受け取る結果
    
- **計算**  
    まだ進行途中で、前へ進める必要がある処理
    
- **実行単位**  
    実行基盤の上で管理される仕事の単位
    
- **ハンドル**  
    外部資源への操作口
    
- **実行基盤**  
    監視、調停、再開を支える土台
    
- **責任スコープ**  
    どこまでを一つのまとまりとして待ち、失敗を集約し、停止を伝えるか
    

**実装者**  
この六つがあると、かなり強いです。  
あるAPIを見たときに、  
「これは値なのか」  
「計算なのか」  
「仕事単位なのか」  
「I/O ハンドルなのか」  
「基盤なのか」  
「責任スコープを作るものなのか」  
を順番に見られます。

**初学者**  
たしかに、この棚がないと、名前だけで「似ている」「違う」を判断してしまいますね。

## ブラウザは何が前面に出ているのか

**初学者**  
では、まずブラウザからお願いします。  
一番なじみがあります。

**解説者**  
ブラウザでは、イベント、`Promise`、`async` / `await`、タイマー、`fetch` などが前面に出ます。  
そのため、**値とイベント駆動の入り口**がかなり強く見えます。

**実装者**  
日常のブラウザJavaScriptでは、  
ユーザー操作がある。  
イベントハンドラがある。  
非同期APIが `Promise` を返す。  
`await` で待つ。  
この流れがとても自然です。  
だから、値の受け取りと出来事への反応が前面に見えやすい。

**初学者**  
逆に言うと、計算とか task とか runtime は見えにくいですね。

**解説者**  
その通りです。  
見えにくいから存在しない、ではありません。  
ただ、ブラウザではそれらが日常APIとして強く露出しません。  
結果として、「イベントが起きる」「`Promise` が返る」「`await` で待つ」という表面の書き味が中心になります。

### 整理

- ブラウザでは、イベントと将来値が前面に出やすい
    
- `Promise` と `await` によって、結果の受け取りは見やすい
    
- 一方で、実行単位や基盤は裏側に隠れやすい
    
- そのため、非同期処理を「イベント + Promise」として理解した気になりやすい
    

## Node.js はブラウザと何が違うのか

**初学者**  
Node.js は同じ JavaScript でも、かなり印象が違います。  
でも、どこが違うのかを言葉にしにくいです。

**解説者**  
Node.js では、ブラウザよりも **ハンドルと実行基盤の気配** が強くなります。  
ソケット、ストリーム、ファイルI/O、サーバー処理など、外部資源を扱う機会が増えるからです。

**実装者**  
ブラウザだと、ユーザーイベントや画面更新が大きい。  
Node.js だと、接続、読み書き、バックグラウンド処理、サーバー側の同時進行が大きい。  
そのため、ハンドルやイベントループ的なものを意識する場面が増えます。

**初学者**  
ただ、それでも `Promise` と `await` がかなり前にありますよね。

**解説者**  
はい。  
そこがNode.jsの特徴でもあります。  
I/O ハンドルの世界と、`Promise` / `await` の書き味が同じ言語の中で密接につながっている。  
そのため、ブラウザよりは基盤の気配が強い一方で、やはり多くの責務が圧縮されて見えます。

### 整理

- Node.js では、ソケットやストリームなどのハンドルが前面に出やすい
    
- 同時に、`Promise` / `await` の高水準な書き味も強い
    
- そのため、ハンドル、継続、基盤が一つの言語体験に圧縮されやすい
    
- ブラウザより基盤を意識しやすいが、それでも責務は混線しやすい
    

## Rust は何が見えやすいのか

**初学者**  
Rust に行くと、急に部品が増えた感じがするんですよね。  
`Future`、runtime、spawn、join、task みたいな話が前に出てきます。

**解説者**  
それは、Rust では **計算と実行基盤の境界が見えやすい** からです。  
JavaScript では圧縮されていたものが、Rust では別々の顔で現れやすい。

**実装者**  
特に `Future` を計算として見る感覚が前に出ます。  
`Promise` 的な「結果を待つ窓口」よりも、「まだ進行中で、進める必要がある計算」が前に見えやすい。  
すると、自然に「誰が進めるのか」という問いが出てきます。  
そこで runtime や executor の存在が見えやすくなるわけです。

**初学者**  
だから、JavaScript の感覚で入ると、急に低レベルになったように感じるんですね。

**解説者**  
そう感じやすいです。  
ただ、実際には概念が増えたというより、**隠れていた責務が露出している**と見たほうがよいです。

### 整理

- Rust では、計算としての `Future` が見えやすい
    
- そのため、実行基盤や task の存在も前面に出やすい
    
- JavaScript で圧縮されていた責務が、別々の部品として現れやすい
    
- 低レベル化というより、責務の露出度が高いと見るほうが理解しやすい
    

## Python はどこが橋渡しになっているのか

**初学者**  
Python はどうでしょう。  
JavaScript と Rust の中間っぽい印象もあります。

**解説者**  
その見方はかなり有効です。  
Python は、`async` / `await` による高水準な書き味を持ちながら、task や event loop や task group 的な概念も比較的見えやすい。  
その意味で、**書き味と責務分解の橋渡し**として見ることができます。

**実装者**  
JavaScript に近い書き心地もあります。  
でも同時に、仕事単位や責任スコープを意識させるAPIも見えやすい。  
だから、構造化並行性や失敗の集約やキャンセル伝播を学ぶ入口としても扱いやすいです。

**初学者**  
なるほど。  
JavaScript 的な `await` の感覚を持ちつつ、task や責任スコープの世界にも入りやすいんですね。

**解説者**  
はい。  
この本では、Python を「中間」ではなく、**責務の地図を見せやすい環境の一つ**として見ます。

### 整理

- Python は `async` / `await` の高水準な書き味を持つ
    
- 同時に、task や責任スコープの概念も見えやすい
    
- JavaScript から構造化並行性へ進む橋渡しとして理解しやすい
    
- 書き味の親しみやすさと責務の見えやすさを両立しやすい
    

## 同じ名前でも責務が違うことがある

**初学者**  
ここまで聞くと、逆に名前が危ないですね。  
同じような名前でも、置かれている責務が違いそうです。

**解説者**  
その通りです。  
この章で特に避けたいのは、**名前が似ているから同じ責務だと思うこと**です。

**実装者**  
たとえば `await` がある。  
それだけで、同じ実行モデルだとは言えません。  
イベントループという言葉がある。  
それだけで、同じ露出度や同じ責務分担だとも言えません。  
`task` という単語がある。  
それだけで、寿命管理や親子関係が同じとも限りません。

**初学者**  
つまり、名前から責務を推定するのではなく、責務の位置から名前を見るわけですね。

**解説者**  
はい。  
その順番が大事です。  
まず「これは値か、計算か、仕事か、基盤か」を見る。  
その後で、その環境ではそれを何と呼んでいるかを確認する。  
この順序なら、かなり混乱が減ります。

## 逆に、名前が違っても責務が近いことがある

**初学者**  
名前が同じでも危ない。  
ということは、名前が違っていても近いものがあるんですね。

**解説者**  
その通りです。  
むしろ、その見方がこの章の核心に近いです。  
ある環境では結果の待機として見えているものが、別の環境では計算の側から見えているかもしれない。  
ある環境では基盤が隠れているが、別の環境では前面に出ているかもしれない。  
この違いを、責務のマップで吸収します。

**実装者**  
だから翻訳するときは、「単語対応表」だけでは弱いんです。  
必要なのは「責務対応表」です。  
この概念は、こっちの世界ではどの棚に置かれているか。  
そこがわかると、別の言語のドキュメントもかなり読みやすくなります。

### 整理

- 名前が同じでも責務が違うことがある
    
- 名前が違っても責務が近いことがある
    
- したがって、単語対応より責務対応のほうが重要である
    
- まず棚を決め、その後で名称を対応づけるほうが混乱しにくい

## どこで読み間違えやすいのか

最後に、ここでの典型的な読み間違いを整理したいです。

**初学者**  
ここまでの章が全部このためにあったんだ、とやっと実感できました。
