---
title: "Promise は何者で、何をしていないのか"
---

**初学者**  
ここまで読んでくると、だんだん `Promise` の見方も怪しくなってきました。  
普段いちばん触っている非同期の道具なのに、もしかしてかなり雑に理解していたのかもしれません。

**解説者**  
その感覚はよい入り口です。  
この章では、`Promise` を否定するのではなく、役割をはっきりさせます。  
JavaScript ユーザーにとって `Promise` はとても身近です。  
でも身近だからこそ、非同期処理そのものと一体化して見えやすい。

**実装者**  
実際、日常のコードでは `Promise` を返して `await` して終わり、という書き方が多いです。  
それで十分に仕事が進むので、「非同期 = Promise」と思いやすいんです。

**初学者**  
かなりそう思っています。  
`Promise` が非同期処理の本体で、`await` がそれを待つ方法、くらいの理解です。

**解説者**  
この章の中心はそこです。  
**`Promise` はとても重要だが、非同期処理の全部ではない。**  
まずはこの一点を、言葉を分けながら整理していきます。

## Promise はなぜ強く見えるのか

**初学者**  
そもそも、どうして `Promise` はこんなに中心に見えるんでしょう。

**解説者**  
理由は単純です。  
日常の JavaScript では、非同期処理の入口にも出口にも `Promise` が出てくるからです。  
API は `Promise` を返す。  
呼び出し側は `await` する。  
失敗も `catch` や `try` / `catch` で扱える。  
複数まとめたいときは `Promise.all` がある。  
かなり多くの場面で、`Promise` が表側に現れます。

**実装者**  
しかも、書き味がきれいです。  
コールバックをネストしなくてよい。  
値の受け渡しもエラー処理も、ある程度そろった形で扱える。  
そのため、`Promise` は単なる部品というより、「非同期の標準形」に見えやすいです。

**初学者**  
便利さがそのまま中心性に見えるんですね。

**解説者**  
そうです。  
でも、便利であることと、責務を全部持っていることは別です。

## Promise はまず「将来の値」として見る

**初学者**  
では、最初の定義はどう置けばいいですか。

**解説者**  
まずは、`Promise` を **将来利用可能になる値** として見るのが基本です。  
いまはまだ手元に完成した値がない。  
でも、後で成功するか失敗するかの形で結果が決まる。  
その結果を受け取るための器として `Promise` を見る。  
これが、この本での最初の置き方です。

**初学者**  
「処理」ではなく「値」なんですね。

**解説者**  
ここが最初の大事な区別です。  
もちろん `Promise` は処理と一緒に語られます。  
でも、概念として最初に置きたいのは「結果の受け取り口」です。  
何かの仕事が終わったあとで得られる結果を、いまの時点から扱いやすくするためのものです。

**実装者**  
言い換えると、いま計算中のものそのものを直接持っている、というより、**あとで受け取る結果に先回りして窓口を持っている**感じです。  
その窓口に、後で成功値か失敗理由のどちらかが届く。  
そう考えると整理しやすいです。

**初学者**  
なるほど。  
「処理の本体」ではなく、「処理が終わったあとに受け取る結果への窓口」と見るわけですね。

**解説者**  
その理解でよいです。  
だから本書では、まず `Promise` を「将来の値」として置きます。  
そのほうが、後で `Future`、task、runtime といった別の概念を分けやすくなるからです。

### 整理

- `Promise` は、いま未確定だが後で決まる結果を表す
    
- 成功か失敗か、どちらかの形で最終状態を持つ
    
- 呼び出し側は、その結果に対して待機や連結を行える
    
- したがって `Promise` は、まず「将来の値」として捉えると整理しやすい
    

**初学者**  
ここで少し視界が変わりました。  
今までは「非同期の処理オブジェクト」くらいに見ていましたが、結果の受け取り口と見るほうが確かに整理しやすいです。

## await は Promise を進めているわけではない

**初学者**  
では `await` はどう位置づければいいでしょう。  
ふだんは `Promise` とセットで考えてしまいます。

**解説者**  
`await` は、`Promise` を進めているわけではありません。  
`await` は、**その結果が使えるようになるまで、その場の続き方を調整する書き方**です。

**初学者**  
前の章でも似た話が出ましたね。  
`await` 自体が仕事の主体ではない、と。

**解説者**  
その通りです。  
ここでも同じ区別が効きます。  
`await` は、将来値を受け取るための待機の記法です。  
何かが裏で進み、結果が確定したときに続きを再開できるようにする。  
でも「何がどう進むか」そのものを、`await` が決めているわけではありません。

**実装者**  
だから、`await` を見て「これで非同期処理の実行モデルが全部わかった」と思うと危険です。  
`await` は接続の書き味をよくしてくれますが、監視、実行、スケジューリング、寿命管理まで説明してくれるわけではありません。

**初学者**  
なるほど。  
`Promise` が将来の値で、`await` はその値を受け取るための待機の書き方。  
ここまでで、だいぶ責務が分かれてきました。

## Promise が得意なこと

**初学者**  
ここまで聞くと、`Promise` の役割はかなり限定的に見えてきます。  
でも実際にはすごく便利ですよね。何がそんなに強いんでしょう。

**解説者**  
その問いは大事です。  
役割を限定して理解することと、価値を小さく見ることは別です。  
`Promise` はかなり強力です。

### 整理

- 将来の成功値と失敗を統一的に扱える
    
- 結果に対して後から処理を接続できる
    
- `await` によって同期風の読みやすい記法で扱える
    
- 複数の結果をまとめる手段を持てる
    
- 非同期APIの表面を一定の形にそろえやすい
    

**実装者**  
特に大きいのは、失敗も結果の流れの一部として扱えることです。  
昔のコールバック中心の書き方では、成功と失敗の扱いが崩れやすかった。  
`Promise` はそこをかなり整えてくれました。

**初学者**  
たしかに、`try` / `catch` である程度自然に書けるのは大きいです。

**解説者**  
はい。  
`Promise` の重要性はそこにあります。  
非同期処理を、値の受け渡しの流れとして扱いやすくした。  
これが JavaScript における大きな前進でした。

## Promise がしていないこと

**初学者**  
では逆に、`Promise` がしていないことは何ですか。  
この章の題名からすると、そこが本題ですよね。

**解説者**  
その通りです。  
`Promise` を正しく使うには、何ができるかだけでなく、何を担当していないかをはっきりさせる必要があります。

### 整理

- `Promise` 自体は、実行基盤そのものではない
    
- `Promise` 自体は、外部資源を直接扱うハンドルではない
    
- `Promise` 自体は、task の寿命管理装置ではない
    
- `Promise` 自体は、キャンセルの責務を単独で完結させない
    
- `Promise` 自体は、複数の仕事の責任スコープを十分に表さない
    

**初学者**  
かなりいろいろしていないですね。

**解説者**  
はい。  
`Promise` は重要ですが、万能ではありません。  
たとえば `Promise` はソケットを監視しません。  
イベントループの代わりにもなりません。  
どの仕事をどこまで一つのまとまりとして管理するかも、単独では十分に表せません。

**実装者**  
ここで設計が崩れやすいんです。  
`Promise` がきれいなので、タイムアウトもキャンセルも親子関係も失敗の集約も、全部 `Promise` の周辺でなんとかしようとしてしまう。  
でも、そこには本来、別の責務が混ざっています。

## Promise は「計算そのもの」なのか

**初学者**  
ここで少し気になることがあります。  
`Promise` を返す関数を書くと、どうしても「処理そのもの」を返している感じがします。  
それでも `Promise` は値なんですか。

**解説者**  
よい疑問です。  
JavaScript では、`Promise` が「処理と密着して見える」のが自然です。  
だから実感としては、計算そのもののように見えます。  
ただ、この本ではあえて一歩引いて、**結果に接続する側**として見るほうを優先します。

**実装者**  
JavaScript では、`Promise` を返す関数を書いた時点で、**計算の開始** と **結果の窓口** が密着して見えやすいんです。  
だから「処理そのものを返している」と感じやすい。  
でも責務分解のためには、その二つをいったん分けて見たほうがよいです。

**初学者**  
つまり、JavaScript の日常感覚としては処理っぽく見える。  
でも責務分解のためには、まず将来値として見るほうがよい、ということですね。

**解説者**  
その通りです。  
これは単なる言い換えではなく、設計上の視点の置き方です。  
`Promise` を最初から「計算そのもの」として扱うと、後で `Future` や task や runtime を区別しにくくなります。

## Promise.all は何をまとめているのか

**初学者**  
`Promise.all` も気になります。  
これって、複数の非同期処理をまとめてくれる便利な道具ですよね。  
かなり万能に見えます。

**解説者**  
便利です。  
ただし、ここでも「何をまとめているのか」を正確に見たほうがよいです。  
`Promise.all` は、複数の **結果の待機** をまとめています。

**初学者**  
結果の待機。

**解説者**  
はい。  
複数の `Promise` に対して、全部の結果がそろうまで待つ。  
その意味では、結果の集約には強いです。  
でも、「複数の仕事を一つの責任スコープとしてどう管理するか」という問いには、それだけでは十分ではありません。

**実装者**  
たとえば、どれか一つが失敗したら残りをどう扱うのか。  
途中で全体を中止したいときに、下流へどう伝えるのか。  
子の失敗を親の責任としてどう束ねるのか。  
こうした話になると、`Promise.all` だけでは足りない場面が出てきます。

**初学者**  
なるほど。  
`Promise.all` は「まとめて待つ」ことはできるけど、「一つのチームとして管理する」ところまでは自動ではやってくれないんですね。

**解説者**  
その理解でよいです。  
この違いは、後で構造化並行性を学ぶときにとても重要になります。

### 整理

- `Promise.all` は複数の結果待機をまとめる
    
- 便利だが、それだけで責任スコープを十分に表すわけではない
    
- 「全部待つ」と「全部を一つの単位として管理する」は別の話
    
- 失敗、キャンセル、寿命管理まで考えると追加の考え方が必要になる
    

## Promise の便利さと責務を混同しない

**初学者**  
だんだん見えてきました。  
`Promise` はかなり便利だけど、その便利さのせいで責務が大きく見えすぎるんですね。

**解説者**  
その通りです。  
この章で大事なのは、`Promise` を過小評価しないことと、過大評価しないことの両方です。

**実装者**  
`Promise` は、非同期APIの表面を整え、将来の結果を扱いやすくし、失敗も流れの中に入れてくれます。  
これは非常に大きな価値です。  
でもそれだけで、実行基盤、仕事の単位、スケジューリング、責任スコープの問題まで全部解決したことにはなりません。

**初学者**  
つまり、`Promise` は非同期処理の世界で重要な一部だけれど、地図全体ではない。

**解説者**  
はい。  
この章の結論はそこです。

### この章の核心

`Promise` は、非同期処理そのものの全部ではありません。  
まずは将来の値として捉えることで、`await`、イベントループ、task、責任スコープとの違いが見えやすくなります。  
`Promise` は結果の受け取りと待機の整理には強い一方で、実行基盤や寿命管理まで単独で担うものではありません。

## まとめ

- `Promise` は、まず将来利用可能になる値として見ると整理しやすい。
    
- `await` は `Promise` を進める主体ではなく、結果を待って続きを再開するための記法である。
    
- `Promise` は成功値と失敗を統一的に扱える点で非常に強力である。
    
- しかし `Promise` 自体は、実行基盤、外部資源ハンドル、task の寿命管理、責任スコープのすべてを担わない。
    
- `Promise.all` は複数の結果待機をまとめるが、それだけで責任スコープ全体を表すわけではない。
    

次章では、`Future` は `Promise` の別名ではない、という論点に進みます。  
将来値としての `Promise` から一歩進み、計算、poll、executor、runtime という見方を導入していきます。
