---
title: "JavaScript ユーザーは何を知っていて、何を誤解しやすいのか"
---

**初学者**  
この本の読み方はわかってきました。  
でも、JavaScript ユーザーって、そんなに特別に誤解しやすいんでしょうか。`Promise` も `await` も普通に使っているし、非同期処理には慣れているほうだと思っていました。

**解説者**  
慣れている、というのはその通りです。  
ただし、その慣れ方に特徴があります。  
JavaScript ユーザーは、非同期処理をかなり早い段階から触ります。しかもブラウザでも Node.js でも、最初から高水準のAPIがかなり整っています。

**実装者**  
言い換えると、入り口が親切なんです。  
コールバック地獄の時代を知らなくても、`Promise`、`async`、`await`、`fetch`、イベントハンドラあたりを使えば、かなり実用的なコードがすぐ書けます。だから非同期処理の「利用経験」は積みやすいです。

**初学者**  
それは強みですよね。

**解説者**  
はい。まずそこを確認しましょう。  
この章は、JavaScript ユーザーを否定する章ではありません。  
何がすでに強みなのか、そしてその強みの裏側で、何を誤解しやすいのかを切り分ける章です。

## JavaScript ユーザーがすでに持っている強み

**初学者**  
最初に強みから整理するんですね。

**解説者**  
そのほうが見取り図を作りやすいからです。  
JavaScript ユーザーは、少なくとも次の感覚をかなり自然に持っています。

### 整理

- 非同期処理では、すぐ結果が返らないことがある
    
- 完了したら続きの処理を進める、という考え方に慣れている
    
- 失敗も値の流れと一緒に扱う感覚がある
    
- 並列に複数の処理を進める書き方に触れている
    
- UI や I/O の待ち時間を意識する習慣がある
    

**実装者**  
これは意外と大きいです。  
同期処理しか知らない環境から入った人だと、「待つ」という発想そのものがまず壁になります。  
でも JavaScript ユーザーは、ボタンを押してイベントが起きるとか、ネットワーク通信が終わるのを待つとか、そういう構造に最初から触れています。

**初学者**  
たしかに、結果がすぐ返らないこと自体は自然に受け入れています。  
`await` で待つのも、イベントで後から処理されるのも、そこまで違和感はないです。

**解説者**  
その感覚は大事です。  
非同期処理が怖いものではなく、「時間差のある処理」だと身体で理解している。これは大きな強みです。

## でも、その強みがそのまま誤解の入り口にもなる

**初学者**  
強みがあるなら、そのまま他言語にも入りやすそうです。

**解説者**  
半分は正しいです。  
ただし、JavaScript で自然だった見え方を、そのまま別の言語や別の設計に持ち込むと、かなり危ないところもあります。

**実装者**  
典型例はこれです。  
「非同期処理 = Promise のこと」  
「待つ = await のこと」  
「裏で動かす = Event Loop がなんとかしてくれること」  
この三つが一体化して見えやすい。

**初学者**  
かなり言いそうです。

**解説者**  
それがこの章の中心です。  
JavaScript ユーザーは、非同期処理の入口に強い一方で、**高水準の書き味に概念が圧縮されている**ため、責務の分離が見えにくくなりやすいのです。

## Promise の成功体験が強すぎる

**初学者**  
まずは Promise ですか。

**解説者**  
はい。  
JavaScript ユーザーにとって、Promise は非同期処理の中心に見えやすい存在です。  
実際、かなり便利です。値があとで届くことを表せるし、成功と失敗を扱えるし、`then` や `catch` でも `await` でもつなげられます。

**実装者**  
しかも、ライブラリや標準APIの多くが Promise を返します。  
`fetch` もそうですし、Node.js のさまざまなAPIも Promise ベースの顔を持つことが増えています。  
だから自然に、「非同期処理の標準形は Promise だ」と感じやすい。

**初学者**  
それは誤解なんですか。

**解説者**  
「JavaScript の日常ではそう見える」という意味では正しいです。  
ただし、そこから「非同期処理そのものが Promise である」と考え始めると危険です。

**初学者**  
違うんですか。

**解説者**  
Promise は、非同期処理全体のすべてではありません。  
Promise は、結果の受け取り方を整える仕組みとしては非常に強力です。  
ですが、実際に何が動いているのか、誰が進めているのか、どこまでが一つの仕事なのか、どこでキャンセルや寿命管理をするのか、といった責務は Promise だけでは表しきれません。

**実装者**  
ここで混線しやすいんです。  
Promise が便利すぎるので、計算そのもの、実行単位、スケジューリング、失敗処理の責任範囲まで、全部 Promise が担っているように見えてしまう。

## await が「魔法の待機」に見えやすい

**初学者**  
でも `await` って、実際かなり魔法っぽく見えます。  
書いたら止まって、終わったら続いてくれるので。

**解説者**  
その見え方自体は自然です。  
ただし、ここでも「見え方」と「責務」は分ける必要があります。

**実装者**  
`await` は便利な記法です。  
ですが、`await` 自体が仕事を進めているわけではありません。  
`await` は、ある値が利用可能になるまで、その場の続き方を調整しているだけです。

**初学者**  
その場の続き方を調整するだけ、ですか。

**解説者**  
そうです。  
`await` は「進める人」ではなく、「ここで待って、再開する」という接続の書き方です。  
だから、`await` を知っていることと、非同期処理の実行モデルを理解していることは同じではありません。

**初学者**  
なるほど。  
`await` が便利なのは事実だけど、それを見て「誰が処理を進めているのか」を理解した気になるのは危ないんですね。

**解説者**  
その通りです。  
JavaScript では `await` が前面に出るので、実行基盤の存在が見えにくくなります。  
でも、実際には何かが処理を前に進め、完了を観測し、続きが再開される必要があります。

## ブラウザと Node.js は似ているが、同じではない

**初学者**  
JavaScript なら、ブラウザでも Node.js でも似たようなものではあるんですよね。

**解説者**  
似ています。  
ただし、同じではありません。  
ここで大事なのは、「書き味が似ている」ことと「責務の置き方が同じである」ことを混同しないことです。

**実装者**  
ブラウザでは、UI イベント、タイマー、ネットワーク、描画などが関係します。  
Node.js では、ファイルI/O、ソケット、サーバー処理、libuv ベースのイベント処理などが前面に出ます。  
同じ JavaScript でも、日常的に触る対象が違うんです。

**初学者**  
でも、コード上は似ていますよね。  
イベントを待ったり、Promise を返したり、`await` したり。

**解説者**  
そうです。  
だからこそ、「同じ言語だから同じ構造だ」と思いやすい。  
しかし実際には、見えているAPIの後ろ側にある実行環境や責務分担には差があります。  
この本では、その差を細かい内部実装の網羅としてではなく、**見え方の差**として整理します。

**実装者**  
ここで重要なのは、差を見つけて不安になることではありません。  
むしろ、JavaScript の世界の中ですでに「見え方の違い」があると知ることで、他言語に進んだときも落ち着いて責務を比較できるようになります。

## 「動く」と「責任を持つ」は違う

**初学者**  
ここまで聞いてくると、かなり大事な区別が見えてきました。  
でも、まだ少しふわっとしています。

**解説者**  
では、この章で特に大事な区別を一つ言葉にしましょう。  
それは、**「動く」と「責任を持つ」は違う**ということです。

**初学者**  
どういうことですか。

**解説者**  
たとえば、ある非同期処理が動いているとします。  
でも、その処理について、次の問いは別々です。

### 整理

- その処理はどうやって進むのか
    
- 結果はどこに現れるのか
    
- 失敗は誰が受け取るのか
    
- 複数の処理をまとめる責任はどこにあるのか
    
- 中断やタイムアウトは誰が判断するのか
    

**実装者**  
JavaScript で日常的にコードを書いていると、これらが一つの書き味に見えやすいです。  
Promise を返して、`await` して、必要なら `Promise.all` を使う。  
これでかなりのことができるので、「責任の置き場」を意識しなくても前に進めてしまう。

**初学者**  
でも、設計の話になると、そのままでは足りなくなるわけですね。

**解説者**  
そうです。  
とくに、処理が増えたとき、失敗の扱いが複雑になったとき、キャンセルやタイムアウトが絡んだとき、あるいは他言語の async ランタイムを読むときに、この差が効いてきます。

## JavaScript ユーザーが誤解しやすい代表例

**初学者**  
ここまでの話を、誤解の形でまとめてもらえますか。  
自分がどこで引っかかりそうか確認したいです。

**解説者**  
では代表例を整理します。

### 整理

- **Promise が async の本体だと思いやすい**  
    実際には、結果の受け取り方と実行そのものは分けて考える必要がある。
    
- **await が処理を進めているように見えやすい**  
    実際には、待機と再開の接続を書きやすくしている。
    
- **Event Loop が万能の実行者に見えやすい**  
    実際には、監視、通知、スケジューリング、再開などは分けて考えたほうが整理しやすい。
    
- **ブラウザと Node.js をほぼ同一視しやすい**  
    書き味は似ていても、前提となる対象や責務の見え方には差がある。
    
- **動くコードを書ければ、責務も理解できていると思いやすい**  
    実務では、動作確認と責任分解は別の能力である。
    

**実装者**  
補足すると、これらの誤解は全部、JavaScript ユーザーが怠けているから起きるわけではありません。  
むしろ、JavaScript の抽象化がよくできているから起きます。  
だから、この章でやりたいのは反省会ではなく、見えていなかった前提を丁寧に取り出すことです。

**初学者**  
そこは少し安心しました。  
「知らなかった」のではなく、「うまく隠されていた」という面もあるんですね。

**解説者**  
そうです。  
よい抽象化は、普段は負担を減らしてくれます。  
ただし、設計を学ぶ段階では、隠れていた責務をもう一度見えるようにしなければなりません。

## この本で、JavaScript の経験をどう使うか

**初学者**  
では、この本では JavaScript の経験をどう使っていくんでしょうか。  
いったん全部忘れるわけではないですよね。

**解説者**  
もちろんです。  
むしろ既存の経験を土台として使います。  
ただし、その経験を「そのまま正解」としてではなく、「分解して再利用できる材料」として使います。

**実装者**  
たとえば、`Promise` に慣れていることは強みです。  
でも次の章以降では、それを「将来値の扱い」として見直します。  
`await` に慣れていることも強みです。  
ただしそれを、「待機と再開の接続記法」として見直します。

**解説者**  
Node.js やブラウザでイベント駆動に触れてきたことも強みです。  
ただしそれを、「Event Loop が全部やる」という一枚絵ではなく、監視、通知、実行、再開といった責務に切り分けていきます。

**初学者**  
つまり、いま持っている知識を否定するのではなく、ラベルを貼り替えていく感じですね。

**解説者**  
まさにその表現でよいです。  
この本の学習は、新しい単語を増やすことよりも、すでに知っている現象に対して、より正確なラベルを与えることに近いです。

### この章の核心

JavaScript ユーザーは、非同期処理の入口に強いです。  
しかし、Promise や `await` の書き味が優れているぶん、値、計算、実行基盤、責任スコープの違いが圧縮されて見えやすくなります。  
この本では、その圧縮をほどいて、経験をそのまま他言語にも通用する地図へ変えていきます。

## まとめ

- JavaScript ユーザーは、非同期処理の利用経験を積みやすく、待機や完了の感覚に強い。
    
- その一方で、Promise が async の本体であるかのように見えやすい。
    
- `await` は便利な記法だが、それ自体が処理を進める主体ではない。
    
- ブラウザと Node.js は書き味が似ていても、責務の見え方や対象には差がある。
    
- 非同期処理では、「動くコード」と「責任を持つ設計」を分けて考える必要がある。
    

次章では、混同されやすい代表例として、Stream・Socket・Event Loop を取り上げます。  
似たように見える名前を、主体、ハンドル、実行単位、実行基盤という観点から切り分けていきます。
