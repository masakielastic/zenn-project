---
title: "Stream・Socket・Fiber・Event Loop を同じものだと思わない"
---

**初学者**  
ここから急にややこしくなってきました。  
`Stream` も `Socket` も `Fiber` も `Event Loop` も、非同期処理の話で一緒に出てくるじゃないですか。  
だから、なんとなく「同じグループのもの」だと思っていました。

**解説者**  
それが、この章でほどきたい誤解です。  
たしかに、これらは同じ文脈で出てきやすいです。  
でも、同じ文脈で使われることと、同じ種類のものだということは別です。

**実装者**  
しかも、JavaScript では特に混ざって見えやすいです。  
Node.js だと `stream` も `net.Socket` もよく出てきますし、最近は `async` / `await` を通じて「処理の流れ」も一つの書き味で扱えます。  
そのせいで、データの通り道、I/O の接続口、処理の中断単位、実行の交通整理が、全部同じ棚に入ってしまいやすいんです。

**初学者**  
まさにそれです。  
全部「非同期のための道具」くらいに思っていました。

**解説者**  
この章では、そこを責務で分けます。  
中心命題は一つです。  
**一緒に登場するからといって、同じ種類のものとは限らない**。  
非同期処理を理解するには、まずこの分類感覚が必要です。

## なぜ同じものに見えやすいのか

**初学者**  
でも、どうしてそんなに混ざって見えるんでしょう。

**解説者**  
理由は単純です。  
どれも「処理がすぐ終わらない場面」に関わるからです。  
ネットワーク通信をすると `Socket` や `Stream` が出てきます。  
待機や再開を考えると `Fiber` のような実行単位の話が出てきます。  
複数の仕事を回す話になると `Event Loop` が出てきます。

**実装者**  
つまり、同じ現場に集まってくるんです。  
でも、同じ現場にいるからといって、同じ職種ではありません。

**初学者**  
たしかに。  
工事現場にいる人が全員同じ仕事をしているわけではない、みたいな感じですか。

**解説者**  
よい比喩です。  
この章では、その「職種の違い」を言葉にしていきます。

## まずは分類の軸を置く

**初学者**  
分類の軸、ですか。

**解説者**  
はい。  
いきなり個別の用語に入るとまた混ざるので、先に大枠を置きます。  
オブジェクトや仕組みは、少なくとも次のように分類して考えると整理しやすくなります。

### 整理

- **主体**  
    判断したり、進め方を決めたりするもの
    
- **値**  
    読み取られるデータそのもの
    
- **ハンドル**  
    外部資源に対して操作を行うための窓口
    
- **実行単位**  
    中断・再開されうる仕事の単位
    
- **実行基盤**  
    複数の仕事を監視し、進行を調停する土台
    

**実装者**  
この章では特に、`Stream` と `Socket` をハンドルとして、`Fiber` を実行単位として、`Event Loop` を実行基盤として見ていきます。

**初学者**  
ここで少し驚きました。  
`Stream` と `Socket` はデータのことじゃないんですね。

**解説者**  
そこが最初の重要点です。  
データが流れるからといって、`Stream` 自体がデータではありません。  
`Socket` も、通信内容そのものではありません。  
どちらも、外部のものに対して読み書きを行うための接点です。

## Socket は何者か

**初学者**  
では、`Socket` からお願いします。  
名前は知っていますが、かなりふわっとしています。

**解説者**  
`Socket` は、ネットワーク越しの相手とつながるための操作口です。  
大事なのは、「相手そのもの」ではなく、「相手に対して送受信するための窓口」だという点です。

**実装者**  
コードの感覚で言えば、`Socket` は「この接続に対して読み書きする能力を持ったもの」です。  
接続の状態を持ち、送る、受け取る、閉じる、といった操作の入り口になる。

**初学者**  
ということは、`Socket` は自分で勝手に判断して動くわけではないんですね。

**解説者**  
その通りです。  
ここが大事です。  
`Socket` は操作を持っていますが、主体ではありません。  
何を送るか、いつ待つか、失敗したらどうするかを決めるのは別の場所です。

**初学者**  
「操作を持つ」と「主体である」は別なんですね。

**解説者**  
はい。  
オブジェクト指向の話になると、「メソッドを持っている = 自己判断する主体」と思いやすいのですが、それは強すぎる見方です。  
`Socket` は、受動的だが操作を持つハンドルです。

## Stream は何者か

**初学者**  
では `Stream` はどう違うんでしょう。  
Node.js の `stream` はよく見かけますが、`Socket` とかなり近く見えます。

**解説者**  
近い関係ではあります。  
ただし、注目しているものが少し違います。  
`Socket` は接続という外部資源への窓口として見やすい。  
`Stream` は、読み書きの流れとして見やすい。

**実装者**  
Node.js では、`Socket` が `Stream` 的な顔を持つこともあります。  
だからさらに混ざりやすい。  
でも概念上は、`Socket` はネットワーク接続のハンドル、`Stream` はデータの読み書きインターフェースや流れの抽象化として分けておくと理解しやすいです。

**初学者**  
つまり、`Stream` は「流れるデータそのもの」ではなく、「流れとして扱うための窓口やモデル」なんですね。

**解説者**  
その理解でよいです。  
ここでも重要なのは、`Stream` もまた主体ではないということです。  
自分でスケジュールを決めたり、全体の仕事を調停したりはしません。

**実装者**  
`Stream` は便利なので、つい何でも背負わせたくなります。  
読み込み、書き込み、バッファ管理、エラー処理、接続状態、再試行、タイムアウト、上位プロトコルの状態管理まで入れたくなる。  
でもそれをやると、すぐに God Object 化します。

**初学者**  
ああ、ここで前の章の「動く」と「責任を持つ」の違いが効いてくるんですね。  
`Stream` にいろいろ載せれば動くかもしれないけど、それで責務が正しいとは限らない。

**解説者**  
その通りです。

## Fiber は何者か

**初学者**  
次は `Fiber` ですね。  
これは `Stream` や `Socket` よりもさらに別物っぽいですが、何者なんでしょう。

**解説者**  
`Fiber` は、外部資源の窓口ではありません。  
`Fiber` は、**中断と再開ができる仕事の単位**として見ると理解しやすいです。

**実装者**  
処理の流れの途中で一時停止して、あとで続きを実行できる。  
その意味で、`Fiber` はデータの通り道ではなく、実行のまとまりです。

**初学者**  
すると、`Fiber` は「何かを読むもの」ではなく、「処理の続きそのもの」なんですね。

**解説者**  
はい。  
`Stream` や `Socket` が外部資源へのアクセス口だとすると、`Fiber` はプログラムの側にある実行の単位です。

**初学者**  
でも `async` / `await` を使っていると、処理の流れが途中で止まって再開するので、たしかに近い話として見えてきます。

**解説者**  
そこが混ざりやすい理由です。  
I/O を待つ場面で `Stream` も `Socket` も `Fiber` も関わることがある。  
でも、  
何を待っているのか、  
何に対して操作しているのか、  
どの単位で再開するのか、  
は別の話です。

**実装者**  
だから、`Fiber` を I/O オブジェクトの一種だと思わないことが重要です。  
`Fiber` は接続でも流路でもなく、処理の継続を保持する側の仕組みです。

## Event Loop は何者か

**初学者**  
では `Event Loop` はどう位置づければいいでしょう。  
これが一番なんでも屋に見えます。

**解説者**  
`Event Loop` は、少なくとも `Stream` や `Socket` や `Fiber` と同列には置かないほうがよいです。  
`Event Loop` は、**複数の仕事の進行を調停する実行基盤**として見るのが基本です。

**実装者**  
くだけて言えば、交通整理です。  
どのイベントが来たか、どの仕事を再開するか、何を今進めるかを回していく。  
ただし、ここでも注意があります。  
`Event Loop` が「全部を一人でやる超主体」だと思うと、また混乱します。

**初学者**  
またそこでも、見え方と責務を分ける必要があるんですね。

**解説者**  
そうです。  
`Event Loop` は、監視、通知、キュー処理、再開のきっかけ作りなどに関わります。  
でも、読み書きの窓口そのものではありませんし、個々の処理内容そのものでもありません。

**初学者**  
つまり、`Event Loop` は `Socket` ではないし、`Stream` でもないし、`Fiber` でもない。

**解説者**  
はい。  
もっと言えば、`Event Loop` はそれらを同じ棚で比較する対象ではなく、  
それらが関わる仕事の流れをどう前に進めるかという、より上位の土台です。

## 4つを並べて見直す

**初学者**  
ここまで来ると、だいぶ違いが見えてきました。  
でも一度並べて整理したいです。

**解説者**  
では、この章の中心をそのまま表に近い形で言い換えます。

### 整理

- **Socket**  
    ネットワーク接続という外部資源に対するハンドル
    
- **Stream**  
    読み書きの流れを扱うためのハンドルや抽象化
    
- **Fiber**  
    中断と再開が可能な実行単位
    
- **Event Loop**  
    複数の仕事を監視・調停して進める実行基盤
    

**実装者**  
この整理の利点は、責務がずれにくくなることです。  
たとえば `Stream` にスケジューラの責務を載せない。  
`Fiber` に I/O ハンドルの責務を載せない。  
`Event Loop` を単なるデータ通路だと思わない。  
この区別だけでも、設計はかなり安定します。

**初学者**  
今までの自分は、  
`Stream` はなんとなく非同期そのもの、  
`Event Loop` はなんとなく裏側の魔法、  
くらいの理解でした。  
かなり雑でしたね。

**解説者**  
雑というより、日常使用ではそれでも足りていたんです。  
この本は、その日常理解を設計に耐える形へ引き上げるための本です。

## なぜこの分類が他言語でも効くのか

**初学者**  
この章の分類は、JavaScript 以外でも効くんでしょうか。

**解説者**  
そこが重要です。  
この分類は、特定の言語のAPI名に依存していません。  
ハンドル、実行単位、実行基盤という分け方は、Rust や Python を読むときにも役立ちます。

**実装者**  
言語が変わると、名前や露出の仕方は変わります。  
でも、外部資源への窓口なのか、処理の継続単位なのか、全体を回す基盤なのか、という問いは共通です。

**初学者**  
なるほど。  
単語を暗記するというより、「これはどの職種か」を判定する感覚を作るわけですね。

**解説者**  
その通りです。  
そして、この感覚がないまま async の設計に入ると、いろいろな責務が一つのオブジェクトに集まり始めます。  
それが次の混乱の種になります。

### この章の核心

`Stream`、`Socket`、`Fiber`、`Event Loop` は、同じ場面で登場しても同じ種類のものではありません。  
`Stream` と `Socket` は外部資源へのハンドル、`Fiber` は実行単位、`Event Loop` は実行基盤として分けて見ることで、非同期処理の責務が見えやすくなります。

## まとめ

- 一緒に登場するものを同じ種類だと思うと、非同期処理の設計はすぐ混線する。
    
- `Socket` はネットワーク接続に対するハンドルであり、主体ではない。
    
- `Stream` は読み書きの流れを扱うためのハンドルや抽象化であり、実行基盤ではない。
    
- `Fiber` は中断と再開が可能な実行単位であり、I/O オブジェクトではない。
    
- `Event Loop` は複数の仕事を調停する実行基盤であり、個々のデータ通路や接続口とは別物である。
    

次章では、さらに一段進めて、`Event Loop` が実際には何を回しているのかを見ていきます。  
「イベントループが全部やっている」という曖昧な理解をほどき、監視と実行の関係を整理します。
