## まずほどきたい混同は何か

**初学者**  
復習ということは、よくある誤解も一度まとめて見たいです。

**解説者**  
では、混線しやすい場所を大きく三つに分けます。  
全部を細かく並べるより、この三つの束で見たほうが復習しやすいです。

### 整理

- **結果と計算の混同**  
    `Promise` が async そのものに見える、`Future` が `Promise` の別名に見える、`await` が処理を進めているように見える
    
- **ハンドルと基盤の混同**  
    `Stream` が非同期処理そのものに見える、`Event Loop` が全部やってくれるように見える
    
- **動作確認と責任設計の混同**  
    `Promise.all` で責任スコープまで表せるように見える、キャンセルが単なる停止命令に見える、動くコードなら責任配置も正しいように見える
    

**実装者**  
どれも、日常の書き味から見れば自然です。  
だから大事なのは、「そんな誤解をするな」と叱ることではありません。  
**なぜそう見えるのかを理解したうえで、責務の棚に戻すこと** です。

**初学者**  
つまり、この章は「間違い探し」じゃなくて、「混線を地図に戻す章」なんですね。

**解説者**  
その通りです。

## `Promise`・`Future`・`task` をどう置き分けるか

**初学者**  
まず一番混ざりやすいところから整理したいです。  
`Promise`、`Future`、`task` は、結局どう置けばいいんでしょう。

**解説者**  
本書では、こう置くと整理しやすいです。  
`Promise` は **値** 側から見る。  
`Future` は **計算** 側から見る。  
`task` は **仕事** の単位として見る。

**初学者**  
ここが本書の中心線でしたね。

**解説者**  
はい。  
`Promise` は、あとで決まる結果への接続口として見る。  
`Future` は、まだ完了していない計算として見る。  
`task` は、その計算が基盤の上で管理対象になった仕事として見る。  
この順番で分けると混線しにくいです。

**実装者**  
ここで重要なのは、似た文脈に出てくるからといって同一視しないことです。  
`Promise` は結果寄りです。  
`Future` は進行中の計算寄りです。  
`task` は、その計算が実際に進行・待機・再開・完了の対象になる実務単位です。

### 整理

- **`Promise`**  
    将来値、または結果への接続口として見る
    
- **`Future`**  
    進行途中の計算として見る
    
- **`task`**  
    基盤の上で管理される仕事の単位として見る
    

**解説者**  
この区別があると、`await`、`spawn`、`join` の違いも見やすくなります。  
`await` は待機、`spawn` は仕事の起動、`join` は合流です。  
同じ非同期文脈に出てきても、責務は違います。

## `Stream`・`Socket`・`Event Loop` をどう置き分けるか

**初学者**  
次に、`Stream`、`Socket`、`Event Loop` ですね。  
ここもかなり混ざりやすかったです。

**解説者**  
ここでは、`Stream` と `Socket` を **ハンドル** として見る。  
`Event Loop` を **基盤** として見る。  
この切り分けが大切でした。

**実装者**  
`Socket` は接続に対する窓口です。  
`Stream` は読み書きの流れを扱うための窓口や抽象化です。  
どちらも、外部資源やデータ流への接点です。  
一方 `Event Loop` は、複数の仕事の進行を監視・調停する土台です。  
つまり、同じ現場に出てきても職種が違います。

**初学者**  
`Stream` はデータそのものではない。  
`Socket` も通信内容そのものではない。  
`Event Loop` はそれらと同列のオブジェクトでもない。  
この区別でしたね。

### 整理

- **`Socket`**  
    ネットワーク接続という外部資源に対するハンドル
    
- **`Stream`**  
    読み書きの流れを扱うためのハンドルや抽象化
    
- **`Event Loop`**  
    複数の仕事を監視・調停して進める実行基盤
    

**解説者**  
非同期処理では、同じ場所に出てくるものが、同じ棚に入るとは限りません。  
ここを見失うと、`Stream` に責務を載せすぎたり、`Event Loop` を魔法の超主体として見たりしやすくなります。

## なぜ「動くコード」でも壊れるのかを地図に戻す

**初学者**  
ここまでの棚があると、第8章の話も見直せそうです。  
なぜ「動くコード」でも壊れるのか、あらためてどうまとめればいいでしょう。

**解説者**  
答えは明快です。  
**責務と責任スコープが混線しているから** です。  
コードが一回動くことと、責任が正しく配置されていることは別でした。

**実装者**  
前章までに見た破綻例を思い出してください。  
野良タスク。  
例外の取りこぼし。  
キャンセルの不整合。  
`timeout` や `retry` の誤配置。  
`Stream` の God Object 化。  
どれも、文法の失敗というより、所有関係や判断主体の失敗でした。

**初学者**  
つまり、何が壊れているかを責務マップに戻して読むと、問題の種類が見えてくるんですね。

### 整理

- **野良タスク**  
    仕事が責任スコープの外にこぼれている
    
- **例外の取りこぼし**  
    失敗の所有関係が曖昧である
    
- **キャンセルの不整合**  
    責任の伝播経路が設計されていない
    
- **`timeout` / `retry` の誤配置**  
    制御ポリシーと実行機構の境界が壊れている
    
- **`Stream` の God Object 化**  
    ハンドルに複数層の責務が集まりすぎている
    

**解説者**  
ここで大事なのは、どれも「非同期だから難しい」で片づけないことです。  
責務マップに戻せば、何がどの棚を越境しているかが見えます。

## 構造化並行性は何を足してくれるのか

**初学者**  
その混線を減らす考え方として、構造化並行性が出てきたわけですね。

**解説者**  
はい。  
構造化並行性は、並行実行を増やすための派手な技法ではありません。  
**仕事を責任の木として扱うための考え方** です。

**実装者**  
ここで効いてくるのが、`TaskGroup` 的な枠です。  
複数の仕事を一つの責任スコープに入れる。  
失敗は上へ集約する。  
キャンセルは下へ伝播する。  
こうして、仕事の所有関係、失敗の集約、停止の伝達を一つの責任モデルでそろえます。

**初学者**  
前の章の「例外は上へ、キャンセルは下へ」がここでも中心なんですね。

**解説者**  
その通りです。  
構造化並行性は、責任の向きを安定させることで、非構造化並行の破綻を減らします。

### 整理

- **破綻の型**  
    野良タスク、例外の取りこぼし、キャンセル不整合
    
- **整える型**  
    責任スコープ、失敗の上方集約、キャンセルの下方伝播
    
- **支える枠**  
    `TaskGroup` 的な責任スコープ
    
- **補助する見方**  
    `ExceptionGroup` 的な失敗集約
    

**実装者**  
だから構造化並行性は、「便利な待機API」ではありません。  
責任スコープを明示するための設計原理です。

## まとめ

- 非同期処理は、値・計算・仕事・ハンドル・基盤・責任スコープに分けて見ると整理しやすい。
    
- `Promise` は結果寄り、`Future` は計算寄り、`task` は仕事単位として見ると混線しにくい。
    
- `Stream` と `Socket` はハンドル、`Event Loop` は実行基盤である。
    
- 動くコードでも、責任スコープが曖昧なら設計は破綻する。
    
- 構造化並行性は、仕事を責任の木として扱うことで破綻を減らす。
    
- 異なる言語や環境も、単語対応ではなく責務対応で読むと比較しやすい。
    
- 最後に持ち帰るべきものは、責務を問うための問いの型である。
