---
title: "なぜ「動くコード」でも設計は破綻するのか"
---

**初学者**  
ここまでで、だいぶ地図は見えてきました。  
でも逆に気になってきたことがあります。  
非同期処理って、実際には「動いているコード」が先にあることが多いじゃないですか。  
それでも設計が破綻する、というのはどういうことなんでしょう。

**解説者**  
この章は、その問いに答える章です。  
ここまでの章では、値、計算、仕事、基盤、責任スコープを分けてきました。  
では、それを分けずに書くと何が起きるのか。  
結論から言えば、**動作確認が通っていても、責任の置き場が曖昧だと設計は簡単に壊れます。**

**実装者**  
非同期処理では特にそれが起きやすいです。  
なぜなら、短いサンプルや単発の成功ケースでは、たいてい動いて見えるからです。  
`await` できる。  
レスポンスが返る。  
テストも一応通る。  
でも、その裏で、誰が失敗を拾うのか、誰が止めるのか、どこまでを一つの仕事として扱うのかが曖昧なまま残ることが多いです。

**初学者**  
たしかに、非同期処理のバグって、「動かない」より「普段は動くのに、ある条件で壊れる」感じがあります。

**解説者**  
その通りです。  
この章の中心命題は一つです。  
**非同期処理では、正しさは「一回動いたか」ではなく、「責任がどこに置かれているか」で決まる。**  
その感覚を、具体的な破綻パターンで見ていきます。

## 最初の破綻は「誰の仕事か」が曖昧なこと

**初学者**  
まず、いちばん基本の破綻は何でしょう。

**解説者**  
「誰の仕事か」が曖昧なことです。  
同期処理でも責任分解は大事ですが、非同期処理では特に、処理が時間差で進むぶん、親子関係や所有関係が見えにくくなります。

**実装者**  
たとえば、ある関数の中で別の非同期処理を起動したとします。  
その処理は呼び出し元の責任範囲に入っているのか。  
それとも独立した仕事なのか。  
途中で失敗したら誰が知るのか。  
呼び出し元が先に終わったとき、その仕事はどうなるのか。  
このあたりが曖昧なままでも、コードは案外動いてしまいます。

**初学者**  
でも、それってあとでかなり危なそうですね。

**解説者**  
はい。  
その危なさが、次の具体例に現れます。  
そして、ここで見えているのは単なる書き方の問題ではありません。  
**仕事の所有関係が曖昧なこと** が、後の破綻の出発点になります。

## 野良タスクはなぜ危ないのか

**初学者**  
野良タスク、ですか。

**解説者**  
はい。  
ここでは、**誰の管理下にもきちんと置かれていない仕事** をそう呼びます。  
起動されたが、どこまで待たれるのか、失敗を誰が拾うのか、停止を誰が指示するのかが曖昧な仕事です。

**実装者**  
非同期処理では、仕事を起動すること自体は簡単です。  
でも、その仕事が「どこにぶら下がっているのか」を明示しないと、後で追跡不能になります。  
親の処理は終わっている。  
画面遷移も済んでいる。  
でも裏で何かがまだ動いている。  
しかも失敗しても、表の流れには戻ってこない。  
こういう状態が起きます。

**初学者**  
たしかに、「ちょっと裏で投げておく」が積み重なると怖いですね。

**解説者**  
野良タスクの問題は、単に数が増えることではありません。  
**責任スコープの外に仕事がこぼれること** が本質です。  
すると、例外もキャンセルも寿命管理も、あとから説明できなくなります。

### 整理

- 野良タスクは、起動されたが責任スコープに適切に結びついていない仕事
    
- 一見便利でも、待機、失敗回収、停止指示の責任が曖昧になる
    
- 親の処理が終わっても裏で動き続けることがある
    
- したがって、単に「起動できる」ことと「管理できる」ことは別である
    

**解説者**  
つまり、野良タスクの破綻は、実行そのものより **所有関係の不在** にあります。

## 例外の取りこぼしは、なぜ起きるのか

**初学者**  
野良タスクの話を聞くと、すぐ例外の問題が思い浮かびます。  
失敗したらどうなるんでしょう。

**解説者**  
そこが次の破綻です。  
非同期処理では、例外が「いまこの場で投げられる」とは限りません。  
後で起きる。  
別の仕事の中で起きる。  
別のタイミングで表面化する。  
だから、どこで拾うかが曖昧だと簡単に取りこぼします。

**実装者**  
同期処理なら、呼び出した関数の周辺で `try` / `catch` すればかなり追いやすいです。  
でも非同期処理では、起動した仕事が後で失敗することがあります。  
その仕事が親の管理下にいないと、例外は「誰にもきちんと所有されていない失敗」になります。

**初学者**  
それで、ログだけ出て終わるとか、エラーハンドラで雑に飲まれるとかが起きるんですね。

**解説者**  
はい。  
大事なのは、例外処理を「その場で捕まえる技法」とだけ見ないことです。  
非同期処理では、**失敗をどの責任スコープに属させるか** が核心になります。

### 整理

- 非同期の例外は、呼び出し地点とは別の時刻・別の仕事で起きうる
    
- そのため、単純な `try` / `catch` の感覚だけでは管理しきれない
    
- 失敗を回収できるかどうかは、仕事が責任スコープに属しているかに依存する
    
- 例外の取りこぼしは、文法の問題というより責任分解の問題である
    

**解説者**  
つまり、ここで壊れているのも、文法ではなく **失敗の所有関係** です。

## キャンセルは「止める命令」ではなく「責任の伝播」である

**初学者**  
例外の次は、やっぱりキャンセルですか。

**解説者**  
はい。  
非同期処理でよく誤解されるのが、キャンセルを単なる停止命令だと思うことです。  
でも設計の観点では、キャンセルは **責任の伝播** として見たほうが整理しやすいです。

**初学者**  
責任の伝播。

**解説者**  
たとえば、親の処理が「もうこの仕事は不要だ」と判断したとします。  
そのとき、子の仕事も止めたい。  
さらに、その子が持っているI/O待機やタイマー待機や下位処理にも、その判断が伝わってほしい。  
これは単なるボタンではなく、仕事の木に沿った伝播の問題です。

**実装者**  
ここで責任スコープが曖昧だと、キャンセルはすぐ壊れます。  
親だけ止まる。  
子だけ残る。  
片方は停止したつもりなのに、別の場所ではまだ送信が続いている。  
タイムアウトと手動キャンセルが競合する。  
こういう不整合が起きます。

**初学者**  
「止める」という一語で済ませると危ないですね。  
実際には、どこからどこへ伝えるかを設計しないといけない。

**解説者**  
その通りです。  
キャンセルは、API一つで解決する便利機能ではなく、**仕事の所有関係をどう作ったか** がそのまま現れる場所です。

### 整理

- キャンセルは単なる停止命令ではない
    
- 親の判断を子や下位処理へどう伝えるかという責任伝播の問題である
    
- 責任スコープが曖昧だと、止めたい仕事だけが止まらず、別の仕事だけ残る
    
- したがって、キャンセル設計は所有関係の設計でもある
    

**解説者**  
つまり、キャンセルの破綻もまた、**責任の線が途中で切れていること** から起きます。

## timeout と retry をどこに置くか

**初学者**  
ここまで来ると、`timeout` や `retry` もかなり危なそうに見えてきました。  
これも責務を間違えると壊れますか。

**解説者**  
はい。  
とても壊れやすいです。  
なぜなら、`timeout` も `retry` も、処理の本体ではなく **制御のポリシー** に近いからです。

**初学者**  
制御のポリシー。

**解説者**  
あるリクエストを何秒待つのか。  
失敗したら何回やり直すのか。  
どの失敗は再試行してよく、どの失敗はすぐ上へ返すのか。  
こうした判断は、I/O ハンドルそのものの責務ではありません。  
また、低レベルの計算そのものの責務とも限りません。

**実装者**  
ここでよく起きるのが、`Stream` や `Socket` や個別APIに、タイムアウトとリトライの判断まで押し込んでしまうことです。  
すると、再利用しづらくなります。  
しかも上位で別のポリシーを適用したいときに衝突します。

**初学者**  
たしかに、タイムアウトって通信の性質だけでは決まりませんよね。  
画面表示なのか、バックグラウンド処理なのか、管理画面なのかでも変わりそうです。

**解説者**  
その通りです。  
だから `timeout` や `retry` は、「動かすための細部」ではなく、**どの層が判断するべきか** が重要になります。  
多くの場合、それはユースケースや業務都合に近い上位側で決まる要素です。

### 整理

- `timeout` と `retry` は処理本体というより制御ポリシーである
    
- 低レベルのハンドルや個別APIに押し込みすぎると責務が重くなる
    
- 上位の用途によって方針が変わるため、判断場所を誤ると再利用性が下がる
    
- したがって、制御ポリシーと実行機構は分けて考えたほうがよい
    

**解説者**  
つまり、ここで壊れるのは、機構そのものより **判断主体の置き場所** です。

## Stream が God Object になると何が起きるか

**初学者**  
ここまでの話を聞くと、`Stream` にいろいろ背負わせたくなる危険がよくわかります。  
前の章でも God Object 化の話が出ましたが、もう少し詳しく知りたいです。

**解説者**  
`Stream` は特に危険です。  
なぜなら、読み書きの入口として自然に見えるからです。  
データも通る。  
エラーも出る。  
待機もある。  
接続状態もある。  
すると、そこに何でも載せたくなります。

**実装者**  
よくある崩れ方はこうです。  
`Stream` が、バッファ管理や接続状態だけでなく、再試行判断、タイムアウト、上位プロトコルの状態、ログ文脈、場合によっては寿命管理まで持ち始める。  
こうなると、最初は便利でも、少し複雑なケースで一気に苦しくなります。

**初学者**  
なんでも窓口に見えるから、なんでも押し込みたくなるんですね。

**解説者**  
はい。  
でも `Stream` は、あくまでハンドルや読み書きの抽象化として見るほうが安定します。  
責任スコープの管理やポリシー判断まで抱え込ませると、他の層との境界が壊れます。

### 整理

- `Stream` はデータの通り道に見えるため責務を集めやすい
    
- しかし、接続管理、制御ポリシー、上位プロトコル、寿命管理まで載せると肥大化する
    
- その結果、再利用性、テストしやすさ、責任境界が崩れる
    
- したがって、`Stream` はハンドルとしての責務を中心に保つほうが安定する
    

**解説者**  
つまり、ここで起きているのは便利さの問題ではなく、**層の境界の崩壊** です。

## HTTP/2 のような多重化で破綻が見えやすくなる

**初学者**  
ここまでの破綻って、単純なAPI呼び出しだけでも起こりますよね。  
でも、もっと複雑なプロトコルだとさらに厳しそうです。

**解説者**  
その通りです。  
特に、複数の仕事が一つの接続の上で同時に進むような場面では、責務の混線が一気に表面化します。  
たとえば、複数のリクエストが一つの接続を共有するケースです。

**実装者**  
HTTP/2 のように、一つの接続の上に複数のストリームがぶら下がる状況を考えてみます。  
ここでは、接続全体の状態、個別ストリームの状態、フロー制御、失敗の影響範囲、タイムアウト、キャンセルが別々に存在します。  
これを「なんとなく一つの非同期処理」としてまとめると、すぐ破綻します。

**初学者**  
たしかに、「接続を止める」のと「一つのリクエストだけ止める」のは全然違いますね。

**解説者**  
はい。  
つまり、多重化や親子関係の強いプロトコルでは、  
何がハンドルなのか、  
何が仕事なのか、  
どこまでが責任スコープなのか、  
どこで失敗を集約するのか、  
を分けないと説明できません。  
この章で見てきた破綻が、より大きな形で現れるわけです。

**解説者**  
ここでも結局、壊れているのは非同期そのものではなく、**責任範囲を一つに潰してしまう見方** です。

## 「一回通る」ことと「設計できている」ことは違う

**初学者**  
ここまで聞いて、ようやく題名の意味がわかってきました。  
「動くコード」でも危ないのは、責任の置き場が曖昧なままでも、一回は通ってしまうからなんですね。

**解説者**  
その通りです。  
非同期処理では特に、成功ケースのデモだけだと設計の弱さが見えにくい。  
でも、失敗、遅延、競合、途中終了、複数同時実行が入ると、一気に露出します。

**実装者**  
だから設計を見るときは、  
「動くか」だけでなく、  
「誰が待つのか」  
「誰が失敗を拾うのか」  
「誰が止めるのか」  
「どこまでが一つの責任範囲か」  
を確認したほうがよいです。

### 整理

- 誰が待つのか
    
- 誰が失敗を拾うのか
    
- 誰が止めるのか
    
- どこまでが一つの責任範囲か
    

**初学者**  
なるほど。  
動作確認は必要だけれど、それは責任設計の代わりにはならないんですね。

**解説者**  
はい。  
そして次章では、その責任範囲をより明示的に扱う考え方として、構造化並行性に進みます。

### この章の核心

非同期処理では、コードが一度動くことと、責任が正しく配置されていることは別です。  
野良タスク、例外の取りこぼし、キャンセルの不整合、`timeout` や `retry` の誤配置、`Stream` の God Object 化は、どれも責務と責任スコープの混線として起きます。  
設計を安定させるには、「どう動くか」だけでなく「誰が責任を持つか」を先に分けて考える必要があります。

## まとめ

- 非同期処理では、一回動くコードでも責任の置き場が曖昧なら設計は破綻しうる。
    
- 野良タスクの本質は、仕事が責任スコープの外にこぼれることにある。
    
- 例外の取りこぼしは、文法の問題よりも失敗の所有関係の問題である。
    
- キャンセルは単なる停止命令ではなく、責任の伝播として設計する必要がある。
    
- `timeout`、`retry`、`Stream` の責務を誤配置すると、制御ポリシーと実行機構の境界が崩れる。
    

次章では、こうした破綻を防ぐための考え方として、構造化並行性を扱います。  
仕事をばらばらに増やすのではなく、責任の木として管理するとはどういうことかを見ていきます。
